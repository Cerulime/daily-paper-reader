## 摘要
内存安全错误仍然是 C 和 C++ 中严重漏洞的主要来源。基于指针的净化器比基于位置的方案（如 LLVM 的 ASan）提供更强的保障，但其开销和兼容性限制阻碍了生产环境的应用。我们提出 PTSan，这是一种 LLVM 净化器，通过在每个指针的高位存储对象标识符，并将对象边界存于固定尺寸的运行时表中，使得基于指针的检查变得实用。这种表示以有限的活跃对象预算换取低开销、优化器可见性和对通用硬件的支持。由于标识随指针值传递，常规的 LLVM 数据流就能传播它，无需显式的每指针元数据指令。将元数据查找与检查执行分离，使两者均以 LLVM IR 的形式暴露，从而支持检查提升、消除和合并，以及为兼容性标签剥离进行全函数最小割放置。在可用时，Intel 线性地址掩码可在硬件中消除剩余的剥离操作。在普通硬件上，PTSan 的性能与已发布的最快基于位置净化器相当：在 x86-64 上 SPEC CPU 2017 的几何平均开销为 57.2%（使用 Intel LAM 时为 46.4%），ARM64 上为 54.7%，在应用形态的 LLVM MultiSource 套件（x86-64）上为 31.5%。这大约是之前具有类似指针-对象权威保障的系统已发布的开销百分比的三分之一，同时保持了独立内存安全测试套件所衡量的跨对象、非对象和时序检测覆盖率。其物理内存开销几乎原生，随着内存成本成为首要约束，这一特性对生产部署至关重要。我们还在实际服务器和安全负载上展示了实际可接受的开销。这些结果表明，PTSan 将实用的基于指针的内存安全引入了一个仅需重新编译的净化器部署模型。

## Abstract
Memory safety errors remain the dominant source of severe vulnerabilities in C and C++. Pointer-based sanitizers provide stronger guarantees than location-based tools such as LLVM's ASan, but their overhead and compatibility limitations have constrained production use. We present PTSan, an LLVM sanitizer that makes pointer-based checking practical by storing an object identifier in each pointer's high bits and its bounds in a fixed-size runtime table. This representation trades a finite live-object budget for low overhead, optimizer visibility, and commodity-hardware support. Because identity travels with the pointer value, ordinary LLVM dataflow propagates it without explicit per-pointer metadata instructions. Separating metadata lookup from check enforcement exposes both as LLVM IR, enabling check hoisting, elision, and merging, plus whole-function min-cut placement of compatibility tag strips. Intel Linear Address Masking eliminates the remaining strips in hardware when available.   On stock hardware PTSan runs at parity with the fastest published location-based sanitizer: 57.2% geomean overhead on SPEC CPU 2017 on x86-64 (46.4% with Intel LAM), 54.7% on ARM64, and 31.5% (x86-64) on the application-shaped LLVM MultiSource suite. This is roughly a third of the published overhead percentage of prior systems with similar pointer-object authority guarantees, while preserving the inter-object, non-object, and temporal detection coverage measured by an independent memory safety test suite. Its physical-memory overhead is effectively native, a critical property for production deployment as memory costs become a first-order constraint. We also demonstrate practical overhead on real-world server and security workloads. These results show that PTSan brings practical pointer-based memory safety into a recompile-only sanitizer deployment model.

### 1. 研究动机与核心问题

内存安全错误是C/C++软件中最主要的安全漏洞根源，占严重漏洞的70%左右，并被广泛用于各种攻击。现有的防御方案中：

*   **基于位置的净化器**（如ASan, RSan）虽然性能好、兼容性高，但其安全保证较弱：它们只检查目标地址是否有效，无法检测指向一个合法对象但越界访问到另一合法对象的“跨对象溢出”（inter-object overflow）。此攻击模式在真实漏洞（如CVE-2025-57807）中已有发生。此外，这类净化器通常附带3倍以上的物理内存开销，不符合生产环境中内存为主要成本约束的现实。
*   **基于指针的净化器**（如SoftBound+CETS, CUP）提供了更强的“指针-对象权威”保证，能检测跨对象溢出。然而，它们通过额外的元数据传播指针边界，导致了极高的性能开销（>150%）和兼容性问题，无法在生产环境中部署。

**核心问题**：如何在保持基于指针的强安全保证的同时，实现与基于位置净化器相当的低性能开销和近乎原生的内存开销，从而将一个实用的内存安全方案以“仅重编译”的模式部署到生产环境中。

### 2. 方法论：核心思想与关键技术

PTSan的核心思想是**在指针值本身中紧凑地嵌入对象标识符（Object ID）**，使其随着普通数据流自然传播，从而避免了维护复杂、昂贵的指针元数据传播机制。

#### 2.1 核心表征与机制

*   **标识符指针**：
    *   利用x86-64架构仅使用低48位作为用户地址空间的特点，PTSan将64位指针的高16位用作**对象ID**，低48位用作规范地址。对象ID充当一个索引。
    *   **身份传播**：因为ID是指针值的一部分，任何LLVM IR层面的指针拷贝、算术运算、`phi`节点等数据流操作都会自动携带这身份信息，**无需任何额外指令**。这是PTSan实现低开销的基石。

*   **运行时元数据表**：
    *   运行时维护一个大小固定（~1.2 MiB）的扁平**边界表**，由对象ID直接索引。每个条目存储对应对象的**基地址（base）** 和**范围（span）**。
    *   这种设计避免了昂贵的影子内存映射和多级查找，边界查询仅是一次内存加载。

*   **访问检查**：
    *   PTSan在一个内存访问执行前，插入检查逻辑：从指针提取ID，加载该ID对应的base和span，然后检查 `(ptr + size) - base <= span`，其中`ptr`是解除了高16位ID的指针地址。
    *   这种检查逻辑验证了整个访问范围是否完全落在该指针所授权对象的边界内，从而从原理上杜绝了跨对象溢出。

*   **时序安全**：
    *   当一个对象被释放时（如`free`），其对应的边界表条目被清零。任何持有该对象ID的悬空指针再次访问时，会因为`span`为0而检查失败。
    *   若该ID后续被新的对象复用，则时序安全变为概率性保护。一个来自旧对象的悬空指针只有当其所指地址恰好落在复用此ID的新对象边界内时才会绕过检测。论文威胁模型分析表明，这需要对标识符的精确预测，对现实攻击者而言概率极低。

#### 2.2 关键优化技术

PTSan的性能优势不仅来自于轻量级的检查机制，更关键的是它将元数据加载与检查操作都保留为普通的LLVM IR，从而让LLVM的优化器可以进行调度和优化。

*   **可优化的IR**：由于边界加载与检查是分离的、普通的IR指令，LLVM优化器可以进行**检查提升（hoisting）**（将不变部分移出循环）、**消除（elision）**（对安全访问移除检查）和**合并（merging）**（用一个范围检查覆盖多次连续访问）。

*   **循环检查优化**：
    *   **两阶段流水线**：该优化在LLVM进行循环变换（如向量化）之前，分析出整个循环的标量内存访问范围并记录在元数据中。在最终插桩阶段，即使循环结构已被打散，PTSan也可以利用之前保存的范围信息，在循环预头部（preheader）放置单个范围检查，替代循环体内的每次迭代检查。这是处理数值计算密集型基准测试的关键优化。

*   **ID剥离提升**：
    *   由于系统调用等接口要求规范地址，软件必须在使用指针前剥离其高16位的ID。PTSan将此问题建模为一个**基于最小割（min-cut）的全局优化问题**。
    *   它将每个SSA值的“已剥离”和“已标记”态映射为图中的节点，将转换操作（剥离和重新标记）的成本作为图的边，通过求解最小割来决定在何处插入转换指令，从而最大化地将转换移出热路径（如循环体外）。

### 7. 优点

*   **设计巧妙且优雅**：将对象身份“免费”融入数据流，是解决指针元数据传播问题的极简方案。固定大小的边界表设计同样简洁高效。
*   **深度编译器协同优化**：论文将内存安全检查与编译器优化深度融合，特别是循环两阶段分析和基于最小割的ID剥离，展示了软件-编译器协同设计的巨大潜力。
*   **评估体系极为严谨和全面**：从计算密集型到应用型benchmark，从性能到内存再到独立的安全检测覆盖率，对比基线选择公平，且包含详尽的消融研究，论据非常坚实。
*   **工程实现完整**：提供了涵盖堆、栈、全局、`mmap`的完整LLVM/compiler-rt开源实现，适配x86和ARM双架构，并考虑了对LAM硬件的支持，工程成熟度高。

### 8. 不足与局限

*   **有限的对象ID预算**：核心限制是只能有65,536个（默认）或32,768个（LAM）同时活跃的对象。虽然论文证明了这在多数应用中是足够的，但某些会分配巨量细小对象的程序（如部分SPEC基准）会超出预算，导致安全降级或程序终止。这使得部署前需要评估目标负载的ID需求。
*   **概率性时序保护**：与SoftBound+CETS的确定性时序安全不同，PTSan在ID复用后保护转为概率性，尽管论文论证了其利用难度极高，但这仍是一个理论上的安全边界。
*   **不检测子对象溢出**：PTSan的保护粒度为“分配对象”级别。对于结构体内部字段间的溢出，它无法检测。
*   **代码覆盖边界**：保护仅限于通过LLVM IR插桩的代码。任何手写汇编、尚未适配的第三方库内部的内存操作，以及非插桩代码中对已剥离指针的访问，都落在安全保证之外。
*   **LAM支持受限**：目前硬件加速依赖于未进入主流Linux内核的LAM U48特性，并面临潜在的Spectre攻击面的安全顾虑，距离通用还有距离。
