## 摘要
动态污点分析（DTA）对于 JavaScript 和 Python 等解释型语言需要三种能力：观察宿主运行时操作、维护并行污点状态以及定义污点如何传播。现有系统将这些能力耦合在插桩机制内——源码重写或引擎原生——导致高昂的运行时开销或需要特定于引擎的嵌入。目前尚缺乏一种运行时无关的通用 DTA 抽象，能将污点语义和状态转换与宿主运行时如何观察和执行它们相分离。我们着手开发一个可扩展、高性能且准确的 DTA 引擎。为此，我们引入了与宿主运行时并行的影子虚拟机，用于跟踪多级污点、起源和跨调用上下文。我们设计了 Mystra，一种具有形式化操作语义的声明式污点规范语言。Mystra 被设计为对语言模型友好，并配备了验证器，可实现可信赖的规则自动合成。Mystra 也是首个声明式地表达高阶函数污点转移的语言。此外，Mystra 规则会被提前编译为二进制表示，并在常数时间内完成分发。我们将这一愿景实现为一个名为 Shar 的工具，它包含一个共享的核心引擎，并在三个运行时上进行了实例化：Node.js 和 Chromium 中的 V8（嵌入）、SpiderMonkey（引擎）和 CPython（语言）。在准确性方面，在 SecBench.js（涵盖四个 CWE 类别，共 493 个范围内的 CVE）上，我们的 V8 实例化在补丁版本测试中达到了 95.5% 的召回率，且零误报。在性能方面，在 NodeMedic 的基准测试中，Shar 的运行时开销仅为原生 Node.js 的 1.85 倍，比 NodeMedic-FINE 在相同工作负载下低 22.7 倍，同时在其支持的类别中召回率提高了 33.2%。

## Abstract
Dynamic taint analysis (DTA) for interpreted languages like JavaScript and Python requires three capabilities: observing host-runtime operations, maintaining parallel taint states, and defining how taint propagates. Existing systems couple these capabilities within an instrumentation mechanism -- source-rewriting or engine-native -- either incurring high runtime overhead or demanding engine-specific embeddings. There is yet to be a runtime-independent abstraction of a general DTA that separates taint semantics and state transitions from how a host runtime observes and executes them.   We set out to develop a DTA engine that is extensible, performant, and accurate. To achieve this, we introduce a Shadow Virtual Machine executing alongside host runtimes that tracks multi-level taint, provenance, and cross-invocation context. We design Mystra, a declarative taint specification language with formal operational semantics. Mystra is designed to be language model friendly, and is equipped with validators enabling trustworthy automated synthesis of rules. Mystra is also the first to express higher-order function taint transfer declaratively. Further, Mystra rules are compiled ahead of time to a binary representation and dispatch in constant runtime.   We implement our vision into a tool named Shar, which contains a shared core engine and instantiations on three runtimes: V8 in both Node.js and Chromium (embedding), SpiderMonkey (engine), and CPython (language). Accuracy wise, on SecBench.js (493 in-scope CVEs across four CWE categories), our V8 instantiation achieves 95.5% recall with zero false positives on patched-version testing. Regarding performance, the runtime overhead of Shar is only 1.85$\times$ over vanilla Node.js on NodeMedic's benchmarks, and is 22.7$\times$ lower than NodeMedic-FINE on identical workloads, all the while producing 33.2% higher recall in its supported categories.

## 2. 论文提出的方法论
### 核心思想
- **关注点分离**：将 DTA 分解为（a）宿主运行时通过统一接口输出操作事件；（b）独立的影子虚拟机维护污点状态并依据声明式规则更新状态。
- **形式化基础**：为影子虚拟机和规则语言提供了完备的操作语义，保证分析行为在任何遵守接口的运行时上一致。

### 关键技术细节
- **影子虚拟机状态**：由四部分组成：
  - **影子栈 (S)**：维护调用帧中操作数的污点映射，随宿主调用栈同步伸缩；异常与异步恢复时同步转移。
  - **影子堆 (H)**：以对象地址 ＋ 属性键为索引记录存活对象的字段级污点。
  - **起源 DAG (G)**：记录污点产生与传播的有向无环图，可重建源－汇路径。
  - **规范上下文 (C)**：全局命名存储（键－值），用于跨越文件 I/O 等外部边界传递污点（如 `writeFile` 存储，`readFile` 恢复）。
- **宿主接口（六项能力）**：操作事件、操作数寻址、操作标识、对象生命周期、运行时查询、续延同步。
- **Mystra 声明式语言**（图 4 给出抽象语法，图 5 给出操作语义）：
  - 基础动作：`source`（标记源）、`sink`（检测汇）、`propagate`（传播）、`clear`（清除）、`set`（更新规范上下文）。
  - 高阶函数支持：`inject`（将元素污点注入回调参数）和 `extract`（从回调返回值合并污点），形式化地处理 `Array.map` 等原生高阶函数。
  - 规则可附加守卫（`where` 谓词），如 `tar.is_proto`，实现上下文敏感的检查。
- **编译优化**：所有 Mystra 规则提前编译为二进制 `.tbin` 格式，在预钩子（pre‑hook）和后钩子（post‑hook）阶段按动作类型分派，实现 O(1) 分发，无运行时解析开销。

### 工作流程
1. 宿主适配器将具体操作（解释器字节码或 JIT 代码）转换为 `enter(op, args)` / `exit(op, result)` 事件。
2. 影子虚拟机在进入操作时查找已编译规则，执行 `inject`、`sink` 及部分 `propagate` 等预动作。
3. 宿主执行操作体，其间可有嵌套事件。
4. 退出操作时执行 `extract`、`clear`、`set` 等后动作，弹出影子帧并桥接返回污点。
5. 若触发 `sink` 且源污点可达，则发出警报并附起源 DAG。

## 8. 不足与局限
- **分析范围限制**：仅关注显式流，**不支持隐式流**。文中指出隐式流对此类漏洞检测提升有限，但理论上仍是一个缺失。
- **边界丢失问题**：
  - **字符串分解**：`String.split` 等操作会产生新值且与原输入失去污点关联，此为显式 DTA 的共性局限。
  - **序列化/持久化边界**：跨 `postMessage` 的 `SerializedScriptValue`、跨文件读写的隐式路径依赖、跨 SQL 读写等需要额外编写桥接规则，否则污点会中断。文中对 n8n 的第二阶段攻击检测失败即因此。
- **性能评估场景有限**：性能实验采用短时间、一次性调用工作负载，**未评估持续 CPU 密集型负载**下始终开启插桩的开销；全 Turbofan JIT 集成也未实现。
- **实验偏差风险**：
  - SecBench.js 中的漏洞较老（2017–2022），多为小型单文件包，可能不充分代表现代复杂应用。
  - 与 NodeMedic‑FINE 的对比受限于该工具仅支持两个 CWE 类别，并非在所有类别上比较。
- **工程限制**：尚未提供按需选择性插桩机制，以进一步降低常态开销；`Object.assign` 等操作可能粗粒度合并污点，需更细粒度的字段敏感源标记。
- **缺乏与更广泛工具的对比**：例如未与同样修改引擎的 `Foxhound`、`PanoptiChrome` 等做直接性能/功能对比。
