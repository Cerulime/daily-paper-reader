## 摘要
DRAM内Rowhammer防御在制造时将缓解阈值固定下来，但真实的Rowhammer阈值（TRHD）随运行时温度变化。我们提出“动态Rowhammer阈值管理”，一种防御无关的运行时层，它每周期根据观测到的温度，通过一个带有VRD驱动保护带g的线性T模型，重新为各项防御设置阈值，并将结果通过各项防御的阈值参数投影到SALT-C、PRAC和TRR上。一个解耦的预言机按δ∼N(1,σ)缩放每DIMM的物理TRHD，打破了模型的自洽性。该层将PRAC在85°C时的72次陈旧违规降至零；当σ=0.10时，扫描g使PRAC违规从38.4（g=1.0）降至9.6（g=0.9）。SALT-C从10次标称-静态违规降至2次（动态）再到0次（自举），延迟影响不超过5.1%。TRR受容量限制；该层起到诊断作用。

## Abstract
In-DRAM Rowhammer defenses pin the mitigation threshold at manufacture time, yet the true Rowhammer Threshold (TRHD) varies with runtime temperature. We propose \emph{Dynamic Rowhammer Threshold Management}, a defense-agnostic runtime layer that re-sources each defense's threshold from the observed temperature once per epoch via a linear-$T$ model with a VRD-motivated guardband $g$, projecting the result onto SALT-C, PRAC, and TRR through each defense's threshold parameter. A decoupled oracle that scales physical TRHD per-DIMM by $δ\sim \mathrm{N}(1, σ)$ breaks model self-consistency. The layer drives PRAC's 72 staleness breaches at 85$^\circ$C to zero; at $σ{=}0.10$, sweeping $g$ collapses PRAC breaches from 38.4 ($g{=}1.0$) to 9.6 ($g{=}0.9$). SALT-C drops from 10 nominal-static breaches to 2 (Dynamic) to 0 (bootstrap), at $\leq$5.1\% latency. TRR is capacity-limited; the layer acts as a diagnostic.

### 1. 论文的核心问题与整体含义
- **背景**：Rowhammer 漏洞随着 DRAM 工艺发展愈发严重，真实的行锤阈值（TRHD）已从 2014 年的约 139K 降至现代 DDR4 的 4.8K 以下，DDR5 的防御甚至考虑 500 以下的 TRHD。
- **动机**：现有的 DRAM 内防御（如 TRR、PRAC、SALT-C）在制造时就将缓解阈值固定下来，但实际 TRHD 会随运行时温度升高而显著降低（例如 50–95°C 内，BER 变化可达 3.3×）。高温下防御器仍按低温阈值工作，就会产生“阈值陈旧性窗口”，导致防御在系统变热时无声失效。
- **核心问题**：如何在不重新设计各防御方案的前提下，让多种基于计数器的 Rowhammer 防御能够根据运行时温度动态调整其阈值，从而消除因温度变化引起的防御缺口。
- **整体含义**：本文并非提出一种新的防御机制，而是一个防御无关的运行时层，它通过“重新获取”各防御的阈值参数来修补温度引起的陈旧性漏洞，可与现有多种防御协同工作。

### 2. 方法论
- **核心思想**：构建一个防御无关的运行时层（Dynamic Rowhammer Threshold Management），每过一个 epoch（默认 10 ms），就按当前 DIMM 温度重新计算一次有效 TRHD，并将其投影到各防御的阈值变量上，**仅收紧、不放松**原防御的阈值。
- **动态 TRHD 模型**：  
  `TRHD_eff(t) = TRHD_init × f_temp(T) × g`  
  - `TRHD_init`：制造时标称阈值（实验设为 1000）。  
  - `f_temp(T) = max( f_min, 1 - β_T · (T - T_ref) )`：线性温度模型，`T_ref = 338 K`，`β_T = 1.2×10⁻²/K`，拟合自 SpyHammer 的 BER-温度数据（R²=0.89）；厂商三星因符号相反通过 SPD MID 查找不同的 `β_T`。  
  - `g`：VRD 驱动的防护带（默认 0.9），用于覆盖 `f_temp` 未能捕获的随机阈值波动，以乘法形式引入以避免低/高 TRHD 下的过保护或欠保护。
- **每 epoch 更新流程**（对任何防御相同）：
  1. **采样**：通过 SMBus/I3C 从 SPD 侧温度传感器获取当前 T。
  2. **计算**：按式(1)计算 `TRHD_eff`。
  3. **投影**：将 `TRHD_eff` 映射为具体防御的阈值变量（见下文）。
  4. **写入**：通过新模式寄存器类（MRW）将新阈值写入每 bank。
- **各防御的阈值投影**（直接使用各防御已公开的阈值参数）：
  - **SALT-C**：按 `MaxACT = 2 × TRHD_eff` 维持其安全边界，解出 `APM` 值。
  - **PRAC**：`ATH_PRAC = max(1, TRHD_eff - N_ABO)`，其中 `N_ABO = 4` 为保守的 ABO 宽限期。
  - **TRR**：`ATH_TRR = max(1, ⌊TRHD_eff / 2⌋)`。
- **解耦预言机**：为打破模型自洽性，在模拟时为每根 DIMM 注入一个缩放因子 `δ ∼ N(1, σ)`，使物理 TRHD 与防御使用的 `TRHD_eff` 不同，以评价模型在真实场景下的鲁棒性。

### 3. 实验设计
- **平台与配置**：DRAMSim3 模拟器，DDR4-3200 配置（1 通道、2 排、16 bank、65K 行/bank、256 个子阵列、V_PP = 2.5 V），设定 `TRHD_init = 1000`。
- **攻击负载**：双面 Rowhammer 追踪（两个相距 64 KB 的攻击行交替激活，每 10 个周期一次，共 600K 次激活，运行 7M 个周期）。
- **比较的防御与校准策略**：
  - 防御架构：SALT‑C、TRR（K=4）、PRAC。
  - 校准策略：
    - **Nom‑static**：按 65°C 标称 TRHD（1000）固定阈值。
    - **Worst‑static**：按 85°C 最差情况 TRHD（760）固定阈值（当今可用的保守静态选择）。
    - **Dynamic**：每 1.6M 个周期（≈1ms 在模拟时钟下）按当前温度动态计算阈值（含 `g=0.9`）。
- **环境与扰动**：
  - 两种温度：65°C（标称，TRHD=1000）和 85°C（热态，TRHD≈760）。
  - 解耦标度 σ ∈ {0, 0.05, 0.10}，每一 σ 值使用 5–10 个随机种子。
  - 防护带 g 扫描（0.7–1.0）用于 Pareto 分析。
- **评价指标**：
  - Breach（安全违规）：行在防御缓解前其激活计数超过真实 TRHD 的次数。
  - Mitigation 事件次数、TRR 的 tracker eviction 次数、SALT‑C 的 ABO 数量、平均与尾部读延迟。

### 4. 资源与算力
- **算力需求**：文中未提及任何 GPU 型号、数量或训练时长。整个评估基于 DRAMSim3 模拟器，模型计算极轻（每个 epoch 仅需一次乘法和减法），所需的只是模拟执行时间，没有依赖大规模算力集群。

### 5. 实验数量与充分性
- **实验组数**：
  - 3 种防御 × 3 种校准策略 × 2 种温度，构成主要对比表。
  - 对 σ = 0、0.05、0.10 分别执行多种子实验，重点分析 σ=0 和 σ=0.10 的 breach 变化。
  - g 的 Pareto 扫描（集中在 PRAC 85°C σ=0.10 下）。
  - SALT‑C 的 bootstrap 变体（启动时直接使用稳态阈值）。
  - TRR 的 K 容量扫描（K=4, 8, 16）。
  - 延迟开销深度分析（仅 SALT‑C）。
- **充分性评价**：
  - 覆盖了主流基于计数器的防御类型，对比了静态校准与动态方案，引入了 σ 和 g 的敏感性分析，并考虑了延迟开销。实验维度较丰富，逻辑自洽。
  - 但所有结果均来自单一模拟配置（双面攻击 trace、固定 TRHD_init=1000、短 epoch 数），未测试复杂攻击模式、长期温度斜坡或多 DIMM 热梯度。实验偏向于展示概念有效性，而非大规模生产环境下的统计验证。整体上对评估本身的公平性较好，但普适性受限于仿真和简单负载。

### 6. 主要结论与发现
- 动态阈值管理能够有效关闭温度引起的阈值陈旧性漏洞：
  - **PRAC**：85°C 下 Nom‑static 出现 72 次 breach，Dynamic 降为 0；当注入 σ=0.10 噪声后，g=0.9 可将 breach 从 38.4（g=1.0）降至 9.6。
  - **SALT‑C**：85°C 下 breach 从 10（Nom‑static）降至 2（Dynamic），若启用启动时的稳态阈值自举可降至 0，且延迟开销 ≤5.1%。
  - **TRR**：由于 tracker 容量（K=4）限制，所有策略均无法避免 breach；而当 K 增至 16 时 breach 消失，表明该层可诊断容量瓶颈。
- 防护带 g=0.9 在 PRAC 上形成了 Pareto 最优，能以可接受的缓解次数换取大幅 breach 降低。
- 该层硬件开销极低（每 epoch 计算 <1 ns，阈值写入占 DRAM 带宽 <0.003%），可部署于主机侧（需 JEDEC 扩展）或完全在片内实现，适配现有温度传感器。

### 7. 优点
- **防御无关性**：不改变各防御内部机制，仅通过其公开阈值参数投影，可适配 SALT‑C、PRAC、TRR 及其他基于计数器的防御（如 BlockHammer、Hydra 等）。
- **开销极低**：计算与写入带宽占比极小，延迟影响仅来自 SALT‑C 的 ABO 本身，动态层不引入额外尾部延迟。
- **实验设计的严谨性**：引入解耦预言机和 σ 扰动，避免模型自洽串通；Pareto 分析 g 的选取；多防御横向对比。
- **实用意识**：讨论了片上实现路径、厂商 β_T 查找、V_PP 作为未来缓解选项等工程考量。

### 8. 不足与局限
- **纯仿真，未在真实硅片上验证**：最关键的假设——首次位翻转阈值（HC_first）与温度的线性关系，未经过直接测量，而是从 SpyHammer 的 BER 数据间接推导。这可能是部署前最大的风险点。
- **模型泛化性受限**：
  - 线性温度拟合基于 DDR4 数据，未证明可直接迁移到 DDR5。
  - 仅覆盖 Micron/Hynix/Nanya 的典型正相关，三星的反向 β_T 仅通过 SPD ID 处理，且 SPD ID 在翻新条上可能不可靠。
- **攻击负载的局限性**：仅测试了简单的双面 Rowhammer 样式，未涉及 Blacksmith、ZenHammer 等复杂频率/多攻击行模式，也未模拟多 epoch 热斜坡或 per-row 的 VRT（可变保持时间）噪声。
- **安全假设的脆弱性**：
  - 攻击者若能操控温度传感器读数、或利用 epoch 间采样窗口造成冷采样/热攻击，可能使防御阈值失准。
  - 未涉及 V_PP 作为控制器侧缓解策略的协同工作。
- **部署障碍**：主机侧方案依赖 JEDEC 新增可写阈值模式寄存器，这在当前 DDR5 规范中并不存在；虽然片上实现可绕过，但仍需厂商支持片内表格校准。
