## 摘要
给定一个漏洞修复提交，触发器定位询问哪个具体语句将易受攻击的程序状态转变为具体的危险操作。这个问题比二进制漏洞检测更难，因为答案需要跨过程的因果推理：在相当大比例的真实CVE中，触发语句位于已修补函数之外的多个调用层，超出了静态规则集和模式匹配语言模型的能力范围。我们提出了AutoTrace，一个智能主体流水线，通过逐层探索代码属性图来定位漏洞触发器，其中LLM智能体决定下一步查看哪里，而确定性准入门决定在报告触发器之前需要哪些证据。智能体从不凭借自身权威接受触发器；每个报告的触发器都有来自图的显式证据支持，因此流水线覆盖了过程内和过程间漏洞，而不依赖于无根据的模型判断。在完整的InterPVD基准上，AutoTrace达到了75.0%的VulnHit和80.8%的FuncHit，超越了同一语料库上的先前最先进水平。基于同样的机制，我们构建了SinkTrace-Bench数据集，该数据集将每个漏洞暴露为从攻击者控制的输入通过传播到危险操作的源到汇（S2S）因果链，这些链来自匹配的易受攻击和已修补的程序状态。它包含1,542个经验证器确认的、完全平衡的易受攻击/安全样本，我们根据专家注释审计其标签保真度。在其上对前沿LLM进行基准测试，我们发现即使最强大的模型也难以区分匹配的对，暴露了触发器定位所针对的因果推理差距。工件可在https://github.com/Erroristotle/AutoTrace获得。

## Abstract
Given a vulnerability-fixing commit, trigger localization asks which specific statement turns the vulnerable program state into a concrete unsafe operation. This question is harder than binary vulnerability detection because the answer demands interprocedural, causal reasoning: in a substantial fraction of real-world CVEs the triggering statement lies several call layers outside the patched function, beyond the reach of static rule sets and pattern-matching language models alike. We present AutoTrace, an agentic pipeline that localizes vulnerability triggers by exploring a code property graph layer by layer, with LLM agents deciding where to look next and deterministic admissibility gates deciding what evidence is required before a trigger can be reported. Agents never accept a trigger on their own authority; every reported trigger is backed by explicit evidence drawn from the graph, so the pipeline covers both intra- and interprocedural vulnerabilities without relying on ungrounded model judgment. On the full InterPVD benchmark, AutoTrace reaches 75.0% VulnHit and 80.8% FuncHit, surpassing the prior state of the art on the same corpus. Building on the same machinery, we construct SinkTrace-Bench, a dataset that exposes each vulnerability as a source-to-sink (S2S) causal chain from attacker-controlled input through propagation to the dangerous operation, drawn from matched vulnerable and patched program states. It comprises 1,542 verifier-confirmed, perfectly balanced vulnerable/safe samples whose label fidelity we audit against expert annotations. Benchmarking frontier LLMs on it, we find that even the strongest struggle to separate the matched pairs, exposing the causal-reasoning gap that trigger localization targets. Artifact available at https://github.com/Erroristotle/AutoTrace.

### 2. 论文提出的方法论
AutoTrace 是一个**代理式流水线**，其核心设计原则是分离 LLM 的语义推理与确定性的证据验证。

- **核心思想**：将触发定位形式化为一个在代码属性图上的搜索问题。LLM 代理充当“语义推理器”决定搜索方向，但任何候选触发器必须通过**严格的确定性准入门**，这些门根据代码属性图证据进行裁决，确保没有报告依赖于未经验证的模型判断。
- **关键技术流程（五阶段）**：
  1. **关键变量提取**：一个代理分析补丁 diff，识别出补丁引入、移除或约束的关键变量。
  2. **补丁范围 CPG 与局部切片**：以补丁为中心，按需构建代码属性图（CPG, 基于Joern），并对关键变量进行局部数据流切片。
  3. **代理式跨过程探索**：基于最优优先搜索，代理在一个函数内分析切片，决定是停留在当前函数、降入被调函数、升到调用者，还是追踪返回源，并将新的待探索项加入优先队列。每一步仅处理一个函数，证据跨调用边缝合。
  4. **多层验证**：候选触发器必须通过两层硬性准入门：
     - **层1（数据流门）**：必须存在从关键变量到候选点的显式数据流路径（或层1b的控制依赖回退）。
     - **层4（上下文一致性门）**：候选点必须在代理声称的控制结构中，且能通过别名记忆与调用链连接回补丁函数。

- **SinkTrace-Bench 揭示推理鸿沟**：即使最强的 LLM（准确率59.0%）在该基准上也表现挣扎。主要失败模式是倾向于将带有危险操作片段的代码一律预测为“易受攻击”，而对补丁消除危险的“安全”版本识别率极低（TNR仅7-31%），证明这些模型缺乏对上游修复操作的因果理解。

### 7. 优点
- **方法创新**：巧妙地将 LLM 的语义灵活性（识别项目特定包装器）与确定性 CPG 证据的可靠性相结合，通过“接受纪律”保证了输出的可审计性。
- **问题聚焦且深刻**：清晰定义了触发器定位这一比检测更难、更精细的问题，并形式化地给出了触发语句与见证路径的标准。
- **高价值数据集贡献**：SinkTrace-Bench 首次以**因果链**为第一类标签，并通过**匹配对**设计强制模型进行因果推理而非模式匹配，直接暴露当前大模型的安全推理短板。
- **实验扎实**：在权威标准基准上进行了多维度、多方法的对比，并对内部组件、失败模式进行了细致分析，成本估算清晰。

### 8. 不足与局限
- **CPG 完整性依赖**：流水线的召回率重度依赖 Joern 构建 CPG 的完整性。对深度嵌套指针别名、宏内隐藏的函数、间接调用等，CPG 可能无法解析，导致切片为空，系统只能弃权。这是其输出的完整性上限。
- **特殊漏洞类型处理**：对于资源泄漏（CWE-401等）这类“缺失操作”的漏洞，无法建立前向数据流，门控逻辑需特殊豁免，降低了对此类漏洞的验证门控强度。
- **开销较大**：虽然单次模型成本低，但整体端到端分析延迟高（中位42分钟），使其适用于离线批量分析，而非交互式场景。
- **数据集偏差**：SinkTrace-Bench 的标签由 AutoTrace 自身的验证器生成（“验证器确认”），虽经人工审计，但并非完全独立的专家标注，其精确触发语句的准确率（83.8%）受限于系统自身能力。
- **泛化性有限**：评估局限于 C/C++ 项目及特定 CWE 类别，对 Java/Python 等管理语言或固件场景的有效性未经验证。
