## 摘要
小程序已成为微信等超级应用中轻量级应用部署的主导范式。为支持无缝集成，超级应用提供了 OAuth 机制用于用户登录。然而，第三方开发者在集成基于 OAuth 的身份验证流程时的不当做法可能导致严重的安全缺陷。本文中，我们发现了三种不同于以往基于静态代码研究的运行时 OAuth 身份验证误用类型，攻击者可借此冒充受害者。为评估其现实影响，我们设计并实现了 MINIAUTH，首个能系统化地大规模分析 OAuth 身份验证误用的框架。MINIAUTH 能自动定位小程序的 OAuth 登录页面，动态执行其工作流，并分析运行时行为。这使其能够处理混淆后的小程序，并发现现有方法无法检测的漏洞。将 MINIAUTH 应用于 44,273 个微信小程序和 2,721 个百度小程序，我们发现了 1,834 个误用案例，包括通过暴露的凭证实现客户端身份伪造、以及通过静态或明文标识符绕过身份验证的关键逻辑漏洞。我们的跨平台评估进一步表明，此类误用并非局限于单一生态系统，而是在不同小程序平台中持续出现。我们还发现了百度 OAuth API 中的一个密码学设计缺陷，该缺陷允许暴力破解会话密钥。我们已负责任地将发现的问题披露给开发者和平台，并获得了确认以及分配的 CNVD/CNNVD 编号。这些结果凸显了制定更稳健的开发者指南和增强平台级防护措施的必要性。

## Abstract
Mini-programs have become a dominant paradigm for lightweight application deployment within super apps such as WeChat. To support seamless integration, super apps provide OAuth mechanisms for user login. However, improper integration of OAuth-based Authentication (OBA) flows by third-party developers can lead to critical security flaws. In this paper, we discover three new types of runtime OBA misuses that differ from prior static-code-based studies, enabling attackers to impersonate victims. To assess their real-world impact, we design and implement MINIAUTH, the first analysis framework for systematically analyzing OBA misuse at scale. MINIAUTH automatically pinpoints the OBA login page of a mini-program, executes the workflow dynamically, and analyzes its runtime behaviors. This enables it to handle obfuscated mini-programs and uncover vulnerabilities that existing approaches cannot detect. Applying MINIAUTH to 44,273 WeChat and 2,721 Baidu mini-programs, we uncover 1,834 misuse cases, including critical logic flaws that enable client-side identity forgery via exposed credentials and authentication bypass through static or plaintext identifiers. Our cross-platform evaluation further shows that such misuses are not confined to a single ecosystem but consistently appear across different mini-program platforms. We also identify a cryptographic design flaw in Baidu's OBA APIs that allows brute-forcing of session keys. We responsibly disclosed our findings to the developers and platforms, receiving acknowledgments and assigned CNVD/CNNVD IDs. These results underscore the need for more robust developer guidance and enhanced platform-level safeguards.

### 2. 论文提出的方法论

*   **核心思想**：设计并实现首个动态分析框架 `MiniAuth`。该框架不依赖静态源码，而是模拟用户行为，自动触发并监控小程序的完整 OAuth 工作流，通过分析运行时网络流量中的安全关键参数来检测逻辑漏洞。
*   **关键技术细节与算法流程**：
    *   **系统工作流（图5）**：MiniAuth 的流程分为三个阶段：
        1.  **预过滤 (Pre-filter)**：通过爬取并分析所有小程序的公开隐私声明页面（按监管要求必须声明对敏感数据的采集意图），利用正则匹配筛选出使用 OAuth 的小程序。
        2.  **登录页识别 (Login Page Identifier)**：
            *   **未混淆程序**：基于 CodeQL 进行静态分析，追踪平台 OAuth API（如 `wx.login`, `getPhoneNumber`）的调用链，并将其与渲染层（WXML/SWAN）的 UI 事件（如 `bindtap`）绑定，定位 OAuth 登录入口。
            *   **混淆程序**：由于源码不可读，系统利用所有页面路径必须声明的配置文件（`app.json`），设计了一种 OCR 辅助的深度优先搜索（DFS）遍历算法（算法2）。它从首页和关键词匹配的路由出发，自动模拟点击，并通过识别超级应用弹出的、样式固定的授权对话框（如图6所示）来确定 OAuth 触发页。
        3.  **动态分析器 (Dynamic Analyzer)**：
            *   **自动交互 (Mini-program Automator)**：通过逆向工程，使用 Xposed 插件（微信）或深度链接（百度）注入目标 `AppID` 和页面 `Path` 来启动小程序。随后使用 Appium 和 OCR 插件自动化处理隐私同意弹窗、平台授权对话框等。
            *   **流量监控与漏洞检测 (OBA Traffic Monitor)**：核心检测逻辑基于“上下文格式模板”和“语义名称模板”来匹配网络流量中的敏感字段（如表1所示，如 `session_key`, `UnionID`, 电话号码模式）。根据这些字段在 OAuth 生命周期中的暴露时机和方式，定义了三类运行时误用的检测规则：
                *   **M1（凭证泄露与身份伪造）**：监控 `session_key` 和 `IV` 是否暴露给前端。
                    *   **M1a (身份伪造)**：若这些凭据在包含 `encryptedData` 的认证请求**之前**泄露，则判定前端有能力构造任意身份的加密报文，实现直接身份伪造。
                    *   **M1b (认证后数据伪造)**：若凭据泄露发生在认证请求**之后**，则虽不能伪造当前登录身份，但可解密并伪造后续的平台加密数据。
                *   **M2（静态标识符滥用）**：在 OAuth 流程完成后，检测流量中是否仅凭静态的 `UnionID` 进行身份断言，而缺乏动态的密码学验证。
                *   **M3（明文标识符传输）**：检测 OAuth 流程完成后，敏感数据（如手机号）是否以明文形式在请求中传输，且服务端未将其与平台验证过的标识符绑。

### 6. 论文的主要结论与发现

*   **漏洞普遍性**：OAuth 身份验证误用问题极为普遍。在使用了 OAuth 的小程序中，**13.8% 的微信**和 **45.4% 的百度**的小程序存在至少一种误用。总计发现 **1,834** 个误用案例。
*   **漏洞分类**：
    *   **M1（凭证泄露）**是最常见的类型，其中 M1a 类型可被直接用于身份伪造。
    *   **M2（静态标识符滥用）**和 **M3（明文数据重放）**相对较少，但通常源于更底层的设计缺陷。
    *   金融、医疗、政务等关键领域的小程序均受影响，存在泄露身份证号、健康记录等风险。
*   **跨平台一致性**：安全缺陷具有极强的跨平台传染性。同一服务在不同的超级应用平台（如微信、百度、支付宝、抖音）上会复现几乎完全相同的漏洞，因为后端服务通常是共享的。一个平台的安全短板会危及所有平台的账户。
*   **平台设计缺陷**：百度 OAuth API 存在密码学设计漏洞，其会话密钥 `session_key` 的一部分被用作 IV，导致可被暴力破解，攻击成本估算仅为 2.74 美元。微信也存在类似设计，但因密钥更长，暂未被成功暴力破解。
*   **开发者对抗行为**：部分开发者为绕过平台简单的静态检查，会通过重命名敏感参数（如将 `session_key` 改为 `sk`）来规避扫描，凸显了采用动态分析方法的重要性。

### 8. 不足与局限

*   **动态分析的固有局限**：
    *   **环境依赖与不稳定性**：动态执行依赖于 UI 渲染和网络稳定。部分小程序能检测到 Mitmproxy 中间人代理，从而拒绝加载或终止认证流程。
    *   **访问限制**：对于内部使用、需要人工身份验证或采用额外应用层流量加密的小程序，`MiniAuth` 无法完成端到端分析。这类应用被保守地排除在误报/漏报统计之外。
*   **实验覆盖偏差**：
    *   **平台局限**：尽管进行了跨平台分析，但大规模自动化评测主要集中在微信和百度。对其他平台（支付宝、抖音）的分析仅限于小规模的手工验证。
    *   **数据集偏差**：日益严格的反爬虫机制使得获取完全无偏的大规模数据集变得困难。文中承认了这一点，但认为动态分析与小程序的热更新机制结合，分析的是最新版本，比静态快照更具实用价值。
*   **分析效率与成本**：动态分析耗时较长（平均 2 分钟/个），且依赖 Root 手机、逆向工程等重操作，目前不适于平台方进行实时、轻量级的安全审核。
*   **检测能力局限**：检测完全依赖网络流量分析。如果整个通信过程采用端到端加密（非 TLS，而是应用层加密），或漏洞逻辑不体现在网络参数中，该方法将失效。
