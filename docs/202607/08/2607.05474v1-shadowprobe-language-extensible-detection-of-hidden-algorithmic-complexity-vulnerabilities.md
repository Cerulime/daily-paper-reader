## 摘要
算法复杂度漏洞（ACV）在对抗性输入触发最坏情况执行行为时出现，导致严重的性能下降或拒绝服务状况。一个关键但未被充分探索的来源是影子复杂度：隐藏在外表良性的标准库API中的非平凡计算成本。由于这些成本在调用点不可见，攻击者可以利用它们引发意外的超线性运行时行为。现有的ACV检测器通常依赖模糊测试、符号执行或混合分析，但它们通常特定于语言，需要大量手动工作来构建测试框架，并且依赖繁重的运行时插桩。我们提出了ShadowProbe，一个可扩展且语言可扩展的框架，通过轻量级静态分析、自动重建执行上下文和大语言模型（LLM）辅助的测试生成来发现ACV。ShadowProbe使用结构化的多阶段流水线：它在影子复杂度信号的引导下静态筛选候选函数，从项目级符号重建最小可执行上下文，并合成大小受控的输入以探测最坏情况行为。然后，它使用执行时间测量和稳健的统计增长推断来验证候选，将真正的算法爆发与运行时噪声（如垃圾收集和JIT编译效应）区分开来。我们在WISE基准上评估ShadowProbe，它相对于现有方法持续提高了分析效率。我们进一步将其应用于大规模系统，包括CPython、JDK、Zig、Rustc和vLLM，发现了许多以前未知的ACV，其中许多已被维护者确认并部分修复。这些结果表明，ShadowProbe可以识别跨多种真实世界代码库的隐藏算法风险。

## Abstract
Algorithmic Complexity Vulnerabilities (ACVs) arise when adversarial inputs trigger worst-case execution behavior, causing severe performance degradation or Denial-of-Service conditions. A key but underexplored source is shadow complexity: non-trivial computational costs hidden inside seemingly benign standard library APIs. Because these costs are invisible at call sites, attackers can exploit them to induce unexpected superlinear runtime behavior. Existing ACV detectors often rely on fuzzing, symbolic execution, or hybrid analysis, but they are usually language-specific, require substantial manual effort to construct harnesses, and depend on heavy runtime instrumentation.   We present ShadowProbe, a scalable and language-extensible framework for discovering ACVs through lightweight static analysis, automated reconstruction of execution contexts, and Large Language Model (LLM) assisted test generation. ShadowProbe uses a structured multi-stage pipeline: it statically screens for candidate functions guided by shadow-complexity signals, reconstructs minimal executable contexts from project-level symbols, and synthesizes size-controlled inputs to probe worst-case behavior. It then validates candidates using execution-time measurements and robust statistical growth inference, separating true algorithmic blowups from runtime noise such as garbage collection and JIT compilation effects.   We evaluate ShadowProbe on the WISE benchmark, where it consistently improves analysis efficiency over existing approaches. We further apply it to large-scale systems including CPython, the JDK, Zig, Rustc, and vLLM, uncovering many previously unknown ACVs, many of which have been confirmed and partially remediated by maintainers. These results show that ShadowProbe can identify hidden algorithmic risks across diverse real-world codebases.

## 2. 论文提出的方法论

ShadowProbe 采用**四个阶段的流水线**：

1. **静态分析（Static Screening）**
   - 基于影子复杂度信号快速筛选候选函数，减少分析范围。信号包括：显式循环/递归、调用可能产生隐藏开销的标准库API（如字符串操作、集合操作、正则表达式、序列化等）。
   - 目标：高密度选择潜在ACV函数，同时滤除常量时间或纯循环无关函数。

2. **上下文恢复（Context Recovery）**
   - 构建项目全局符号索引，记录定义、引用、源码范围等。
   - 从目标函数出发，进行**有界依赖闭包展开**，自动恢复所需的数据类型、超类、辅助函数等上下文，产生最小可编译/可执行片段，无需手工编写Harness。

3. **最坏情况输入构造（Worst‑Case Input Construction）**
   - 利用LLM，根据目标函数签名、恢复的上下文和Harness契约，生成多个大小参数化的输入生成器 \(G(n) \to (arg_1, \dots, arg_k)\)，要求输入大小随 \(n\) 线性增长（\(\Theta(n)\)），且生成器本身不能引入额外计算。
   - 通过编译、执行和大小检查反馈自动修复生成器。从多条策略中选择在探测执行中增长最快的生成器作为最终候选。

4. **动态验证（Dynamic Validation）**
   - 以自适应对数采样（算法1）在输入大小 \(n\) 上执行候选，测量壁钟时间。
   - 多次重复采点以抑制噪声。
   - 基于测量点进行曲线拟合，将增长分类为 **低阶（≤O(n log n)）、多项式、指数、未知**，只确认多项式或指数类为ACV。
   - 利用统计推断滤除GC和JIT抖动，保证结果的可复现性。

整个流程不依赖语言特定插桩或仪器，静态分析和符号索引的实现通过 Tree‑sitter 等工具适配不同语言，只需少量解析和提取代码。

## 6. 论文的主要结论与发现

- ShadowProbe 在 WISE 基准上 **100% 正确**识别所有程序的复杂度类别，且平均分析时间比 SPF‑WCA 快 4.7 倍，比 Badger 快 8.1 倍。

- 在五个真实项目中 **新发现 162 个 ACV**，其中 84 个已确认，41 个已修复，22 个修复中。主要集中在解析和文本处理密集的模块（如 CPython 的 argparse、idlelib、xml；JDK 的 security、util、net 包）。
- 根因模式：**不可变性导致的重复拷贝**、**线性开销列表操作在循环内**、**缺乏记忆的递归**等。
- 证实了影子复杂度是长期潜伏的隐患（如 Python singledispatch 自 2013 年引入的指数复杂度，Zig 前端的线性扫描等）。

## 7. 优点

- **语言可扩展**：静态分析和符号抽取只需少量语言特定代码（Python 1735 行、Java 1218 行、Zig 391 行、Rust 596 行），其余均通用。
- **自动化高**：无需手工编写 Harness 或驱动，从源码到可执行测试全自动。
- **轻量且高效**：不使用重量级动态插桩，仅依赖壁钟测量，分析时间远小于符号执行工具。
- **测量驱动验证**：用真实执行时间分类，避免仅靠 LLM 或静态推断的不准确，区分噪声。
- **实际价值高**：发现大量真实 ACV，得到开发者积极响应和修复。

## 8. 不足与局限

- **依赖 LLM**：输入生成阶段需调用外部模型，可能带来成本、延迟和隐私问题；不过论文模型可替换，且测量验证可缓解 LLM 错误。
- **扩展语言需工程工作**：虽然核心通用，但适配新语言仍需实现解析和符号抽取，且表格中的“影子复杂度 sink”列表需语言专家补充。
- **检测范围局限**：主要针对单函数、输入驱动的复杂度；不涉及并发、分布式或协议层的问题。
- **潜在漏报**：静态筛选可能丢弃某些影子复杂度模式（如 281 个筛选中漏掉两个未见过的 sink），上下文恢复受预算限制。
- **实际性限制**：部分发现的 ACV 可能在实际部署中不易触发，或仅被视为性能问题而非安全漏洞，需维护者主观判断。
- **评估基准**：PyEmailBench 仅基于单一模块，代表性有限；与 HypoFuzz 对比时需大量手工适配，结果可能不够公平。
