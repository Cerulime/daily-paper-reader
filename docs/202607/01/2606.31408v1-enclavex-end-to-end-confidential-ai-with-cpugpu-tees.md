## 摘要
大型语言模型 (LLM) 迅速普及，推动了 AI 应用的广泛采用。大多数部署依赖于集中式基础设施，如 Microsoft Azure、Google Cloud 或 AWS，要求用户共享敏感数据和训练或微调代码。这种依赖性引发了重大的安全和隐私担忧，因为必须信任云提供商以确保机密性和完整性。可信执行环境 (TEE)，例如 Intel SGX/TDX、AMD SEV-SNP 和 ARM CCA，已被引入以减轻这些风险。最近，NVIDIA 开发了 GPU TEE（例如 H100/H200），然而对集成 CPU 和 GPU TEE 的端到端工作流程的全面评估仍然有限。关键方面，包括性能开销、远程认证以及 AI/LLM 应用的安全保证，尚未得到充分研究。本文通过提出一个结合 CPU 和 GPU TEE 的端到端工作流程来填补这一空白。我们提出了机制来确保在 VM 级别（通过 Intel TDX 和 AMD SEV-SNP）和应用级别的机密性和完整性，并重点指出了诸如 Kubernetes 管理员能够访问机密 VM 内容等漏洞。最后，我们使用行业基准评估了我们系统的性能开销，重点关注集成 Intel TDX 与 NVIDIA H200 GPU 的配置。

## Abstract
Large Language Models (LLMs) have rapidly proliferated, driving widespread adoption of AI applications. Most deployments rely on centralized infrastructures such as Microsoft Azure, Google Cloud, or AWS, requiring users to share sensitive data and training or fine-tuning code. This dependence raises significant security and privacy concerns, as cloud providers must be trusted to ensure confidentiality and integrity.   Trusted Execution Environments (TEEs) e.g., Intel SGX/TDX, AMD SEV-SNP, and ARM CCA have been introduced to mitigate these risks. More recently, NVIDIA has developed GPU TEEs (e.g., H100/H200), yet comprehensive evaluations of end-to-end workflows that integrate CPU and GPU TEEs remain limited. Critical aspects, including performance overhead, remote attestation, and security guarantees for AI/LLM applications, have not been sufficiently studied.   This paper addresses this gap by presenting an end-to-end workflow that combines CPU and GPU TEEs. We propose mechanisms to ensure confidentiality and integrity at both the VM level (via Intel TDX and AMD SEV-SNP) and the application level, highlighting vulnerabilities such as Kubernetes administrators' ability to access confidential VM contents. Finally, we evaluate the performance overhead of our system using industry benchmarks, focusing on configurations that integrate Intel TDX with NVIDIA H200 GPUs.

### 1. 论文的核心问题与整体含义
*   **核心问题**：云上大规模部署的 LLM 服务要求用户将敏感数据、训练/微调代码上传至中央云平台（如 Azure、AWS），这迫使租户必须完全信任云服务商，引入显著的安全与隐私风险。虽然 CPU TEE（如 Intel TDX）和 GPU TEE（如 NVIDIA H200）已分别出现，但**缺少对 CPU/GPU TEE 联合使用的端到端工作流的系统研究**，包括性能开销、远程证明及完整的安全保障。
*   **额外暴露的薄弱点**：现有方案即使把机密虚拟机（CVM）与机密 GPU（cGPU）对接，仍然不能彻底消除云管理面的威胁。尤其是 **K8s 管理员**仍保留全权 API 权限（如 `kubectl exec`），可以以 root 身分进入机密 VM 直接访问内存或窃取数据。
*   **整体含义**：本文提出 **EnclaveX**，一套集成了 CPU TEE（Intel TDX、AMD SEV‑SNP）、GPU TEE（NVIDIA H200）与 SCONE 运行时的端到端机密 AI 平台。其目标是在不修改 AI 应用的前提下，实现从硬件信任根到应用层的多层防护，并量化此种架构的额外性能代价。

### 2. 论文提出的方法论
**核心思想**：将信任边界从硬件 TEE 延伸到应用进程内部，使其免受 **K8s 管理员（root）** 的窥探。即使管理员已进入 CVM，也无法获得应用解密密钥或 dump 内存。

**关键技术组件与流程**：
*   **Configuration and Attestation Service (CAS)**：运行在自有 TEE 内的信任根，负责管理安全策略、校验证明、颁发密钥。
*   **SCONE 内核模块**：部署在 Guest OS 中，受 CVM 启动度量覆盖。该模块在运行时对 AI 应用进行**动态度量**，生成加密证明（quote），同时**禁用 Guest 内核中的 memory dump 功能**，阻止管理员转储应用内存。
*   **多层远程证明**：
    1.  CVM 层证明：Intel TDX 生成包含固件、Guest OS、内核模块等度量的 quote。
    2.  cGPU 层证明：NVIDIA H200 通过 SPDM 协议协商密钥，并在挑战-应答流程中提供包含 VBIOS、驱动等度量的证明。
    3.  应用层证明：SCONE 运行时度量 ML 应用，由内核模块签名后送至 CAS。
*   **工作流**（见图 1）：
    1.  用户定义安全策略，并先证 CAS。
    2.  提交策略至 CAS，并启动 CVM（附带 VM ID）。
    3.  CVM 启动时度量并证明固件、Guest OS、GPU。
    4.  VM 侧向 CAS 提交 CVM 与 GPU 的证明报告。
    5.  CAS 验证后，向 VM 内的证明模块颁发签名密钥（保证单例性）。
    6.  内核模块度量运行中的 AI 应用，生成应用级 quote，发送给 CAS。
    7.  CAS 验证通过后，下发配置与解密密钥，应用方可执行。
*   **为何无性能开销**：SCONE 运行时以 “SIM” 模式工作，依赖 CVM 的内存加密而非二次加密，系统调用同步执行且不复制参数，不干扰计算路径。

### 3. 实验设计
*   **测试负载**：LLM 推理（Llama‑3.1‑8B‑Instruct，FP8 量化，TensorRT‑LLM 后端，Triton Inference Server）。
*   **Benchmark 指标**：
    *   **吞吐**：Tokens Per Second (TPS)。
    *   **延迟**：Time Between Tokens (TBT，即 Inter‑Token Latency)、Time to First Token (TTFT)。
    *   **证明延迟**：从发起证明到收到验证结果的总时长。
*   **对比模式**：
    1.  **native**：标准虚拟机（非 TDX） + GPU 非 CC 模式。
    2.  **cvm**：Intel TDX 虚拟机 + GPU CC 模式。
    3.  **scone (EnclaveX)**：TDX 虚拟机 + GPU CC 模式 + SCONE 运行时及应用级证明。
*   **参数配置**：
    *   TPS/TBT 测试：输入/输出 token 固定 128，batch size 从 1 扫到 64。
    *   TTFT 测试：batch size = 1，输出 token = 128，输入 token 从 32 扫到 1024。
    *   每项实验发送 500 × batch_size 次请求（例如 batch=64 时 32 000 次请求）。
*   **证明实验**：重复 100 次，分别测量 TDX+GPU 原生的证明延迟和加上 EnclaveX 应用级证明后的总延迟。

### 4. 资源与算力
*   **硬件平台**：Supermicro SYS‑322GA‑NR，配备双路 Intel Xeon 6900 系列处理器、755 GB 内存，以及 **单块 NVIDIA H200 NVL（141 GB HBM3e）**，通过 PCIe 5.0 连接。
*   **计算类型**：全部为**推理**实验，未涉及模型训练或微调。
*   **算力说明**：文内仅给出推理时使用的单张 H200，未提供训练算力或 GPU 小时数。

### 5. 实验数量与充分性
*   **实验组数**：主要包含三组性能对比实验（TPS、TBT、TTFT，每组下多个 batch/input token 设定）和一组证明延迟实验，整体约为 **4 类场景**。
*   **充分性**：
    *   **优点**：针对核心目标“是否引入额外开销”，实验直接、清晰地给出了回答——EnclaveX 在推理路径上无额外性能损失。
    *   **局限**：实验聚焦于单模型、单 GPU、纯推理场景。缺少训练/微调负载评估、更大规模 LLM（70B+）、多 GPU、分布式拓扑以及多租户并发等更具挑战性场景的验证。因此，结论的泛化能力有限。
*   **客观性与公平性**：对比基线设置合理（native vs cvm vs scone），所有测试使用相同的硬件、模型和请求生成器，控制变量清晰。

### 6. 论文的主要结论与发现
1.  **EnclaveX 无额外推理开销**：在 SCONE“SIM” 模式下，其 LLM 推理的吞吐与延迟与原生 CVM 基线完全相同，差异落在标准差之内。
2.  **CVM+CC GPU 存在明显 IO 瓶颈**：相对于非保密原生模式，TDX + GPU CC 的 TPS 下降 35.0%–62.8%，TBT 增加 35.2%–73.9%，TTFT 增加 11.2%–38.3%。这种开销主要来自 cGPU 与 CVM 之间经 bounce buffer 的加密 IO，随批大小和输入 token 增大而降低（GPU 计算时间占比提升）。
3.  **证明开销可忽略**：EnclaveX 的额外应用级证明仅带来约 0.0147 秒的延迟，叠加在原生 CVM+GPU 证明的 1.1611 秒之上，总开销仅 ~1.27%。
4.  **安全提升**：通过应用层度量、密钥绑定与内存 dump 禁用，系统成功将 K8s 管理员排除在应用的信任基（TCB）之外，实现了“管理员可进入 VM 但无法触碰数据”的安全目标。

### 7. 优点
*   **安全层次创新**：首次在 CVM 内为用户进程建立 SGX 风格的进程级 TEE，并将此安全边界延伸至 cGPU，防范 root 权限管理员。
*   **不侵入应用**：基于 SCONE 的封装技术，无需修改 AI 应用代码即可接入 TEE 保护。
*   **硬件架构无关性**：设计上规避了对特定 CPU 指令（如 AMD VMPL）的依赖，可移植到 Intel TDX、AMD SEV‑SNP、ARM CCA。
*   **基准贡献**：首次提供附加完整远程证明链的 H200 保密 LLM 推理性能数据，填补了学术空白。
*   **工程开源化**：证明内核模块计划开源，利于社区复现与改进。

### 8. 不足与局限
*   **场景覆盖狭窄**：仅评估了推理工作流，缺乏对训练、微调或大规模多租户部署的覆盖，难以直接外推至所有 AI/ML 生产负载。
*   **模型与规模单一**：只在一款 8B 参数模型上实验，未验证超大模型（如 70B、135B）或多 GPU 协同下的开销变化。
*   **安全模型受限**：威胁模型明确排除侧信道攻击，而现实中针对 TEE 的侧信道是重要威胁向量；文中虽提及 SCONE 有部分缓解，但未在 EnclaveX 环境下评估。
*   **IO 瓶颈未解决**：cGPU 的 bounce buffer 加密 IO 在低 batch/短 token 时开销极大，且文献中提及的 TEE‑IO 技术（TDX‑Connect）尚未获 NVIDIA 支持，成为当前方案的性能短板。
*   **功能性限制**：禁用 Guest 内存 dump 虽能阻挡管理员，但可能影响故障诊断、调试与运维能力，文中未讨论。
