## 摘要
数据流分析是Android应用隐私与安全审计的基础。近期出现的编码智能体能够通过搜索、阅读和理解仓库代码，协助完成非平凡的源到汇数据流分析任务。然而，当这些任务以批量工作负载执行时，当前的智能体分析设置会产生巨大的重复分析成本。分配给不同污染源的智能体实例可能会检查共享的代码片段，因为目标应用中的代码复用会导致不同的数据流路径汇聚到共享的程序逻辑。由于这些智能体实例是上下文隔离的，对共享代码片段的分析可能会在批次内重复进行，不必要地消耗API预算并限制可扩展性。我们提出FlowArk，一个知识复用系统，通过将已完成分析的知识提供给后续智能体实例，来降低批量智能体数据流分析中的重复分析成本。具体而言，FlowArk将已完成的分析历史提炼为可复用的知识候选，将这些候选打包成可匹配的知识条目，并将匹配到的条目注入到后续智能体实例的上下文中。我们在OpenCode上实现了FlowArk，并在来自50个开源Android应用的4,685个源到汇数据流分析任务上进行了评估。与标准OpenCode相比，启用FlowArk的OpenCode在保持相当分析质量的同时，将端到端API成本降低了26.83%。此外，在100美元的预算下，FlowArk能够多完成36.66%的任务（1,060 vs. 776）。

## Abstract
Data-flow analysis is foundational to Android app privacy and security auditing. Recent coding agents can assist with non-trivial source-to-sink data-flow analysis tasks by searching, reading, and reasoning over repository code. However, when these tasks are executed as a batch workload, current agentic analysis setups incur substantial re-analysis cost. Agent instances assigned to different taint sources may inspect shared code fragments, because code reuse in the target app can cause different data-flow paths to converge on shared program logic. Since these agent instances are context-isolated, analysis of these shared code fragments can be repeated within a batch, unnecessarily consuming API budget and limiting scalability.   We propose FlowArk, a knowledge-reuse system that reduces re-analysis cost in batch agentic data-flow analysis by making knowledge from completed analyses available to later agent instances. Specifically, FlowArk distills completed analysis histories into reusable knowledge candidates, packages these candidates into matchable knowledge entries, and injects matched entries into a later agent instance's context. We implement FlowArk on OpenCode and evaluate it on 4,685 source-to-sink data-flow analysis tasks from 50 open-source Android apps. Compared with standard OpenCode, FlowArk-enabled OpenCode maintains comparable analysis quality while reducing end-to-end API cost by 26.83%. In addition, under a USD 100 budget, FlowArk completes 36.66% more tasks (1,060 vs. 776).

### 2. 论文提出的方法论

- **核心思想**：将知识复用从依赖智能体主动发起的记忆搜索，转变为一种由运行时上下文触发的、基于规则的可复用知识匹配机制。
- **系统架构：FlowArk**
    FlowArk 通过一个三阶段的知识复用循环来实现其目标，该循环在智能体实例完成后和运行时运行。
    - **第一阶段：选择性知识蒸馏**
        - **目的**：从已完成任务的冗长分析历史中，精确提取出可复用的知识候选。
        - **方法**：
            - **生成式蒸馏**：在智能体完成分析后，FlowArk 向其发送一个结构化的蒸馏请求，要求其遵循三个选择规则生成知识候选：
                - `证据支持`：候选知识必须有已读代码证据支撑。
                - `边界清晰`：候选知识需涵盖一个共享代码片段，具有明确的入口、边界、适用条件和后备措施。
                - `可操作`：候选知识需指明哪些代码可以跳过，以及后续应从何处继续分析。
            - **历史重叠引导的边界选择**：通过比较当前任务的数据流路径（方法调用链）与历史任务的数据流路径，识别出在不同任务间反复出现的“重复子路径”。将此子路径作为边界提示，引导知识蒸馏过程聚焦于共享的中间逻辑，排除任务特定的上下游代码。
    - **第二阶段：知识打包与准入**
        - **目的**：将知识候选转化为可在运行时被匹配的知识条目，并过滤掉低质量的条目。
        - **方法**：
            - **知识打包**：为每个知识候选附加两个组件：
                - **匹配规则**：使用稳定的代码锚点（如精确符号、方法调用、包名前缀等）定义该知识在何种运行时上下文中有效。这些规则被组织成`require_any`、`require_all`和`exclude`三个桶，以表达逻辑组合关系。
                - **下游代码提示**：一个可选的、分支级别的注解。当共享代码片段包含参数敏感的路径选择时，它根据当前上下文中的键（如协议值）映射到下游应检查的具体代码位置。
            - **知识准入与修复**：对打包好的知识条目进行检查，验证其是否有具体代码证据、清晰边界、可操作指引以及精确的匹配规则。仅规则层面有问题的候选可进入受限的修复步骤，否则被丢弃。只有通过准入的条目才存入知识库。
    - **第三阶段：上下文感知的运行时知识注入**
        - **目的**：在后续智能体实例即将重复分析共享代码片段之前，及时将匹配的知识注入其上下文，以引导其跳过分析。
        - **方法**：
            - **运行时上下文监控**：在任务分析请求提交时和每次工具调用完成后这两个稳定边界，收集“最新上下文增量”（如任务描述、工具输出文本）作为匹配的输入。
            - **基于规则的知识匹配**：使用每个已准入知识条目的附加匹配规则，轻量级地评估“最新上下文增量”。先检查`exclude`规则，再评估正向规则。只有正向证据满足且无排除规则被触发的条目才会被选中注入。
            - **紧凑知识注入**：将匹配到的知识组装成紧凑的知识块，并根据注入历史避免重复全量注入。只选择与当前上下文相关的下游代码提示。最后，通过智能体接口（如用户输入通道）将知识块注入其上下文。
