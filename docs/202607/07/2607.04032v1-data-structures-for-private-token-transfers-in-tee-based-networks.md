## 摘要
基于可信执行环境（TEE）的机密智能合约网络承诺隐私性，但仍易受存储访问模式攻击，这类攻击可在代币传输中关联发送方与接收方。当合约在传输过程中更新接收方余额时，即使数据已加密，所访问的唯一存储键也会泄露交易关系。本文引入了两种新颖的数据结构来解决这一漏洞：延迟写入缓冲区（DWB）和分桶条目位字典树（BTBE）。DWB通过缓存待处理传输并随机结算条目来延迟接收方余额更新，从而打破传输执行与接收方存储访问之间的直接关联。BTBE通过将地址分组为恒定大小的桶来进一步增强隐私性，阻止泛洪攻击并为余额查询创建匿名集。此外，我们提出了一种隐私通知系统，可为机密合约实现实时、保护隐私的推送通知。我们的领域特定方法利用了代币传输的独特特性——非对称余额更新和对延迟结算的容忍——以实际性能实现概率性匿名保证。

## Abstract
Trusted execution environment (TEE) based confidential smart contract networks promise privacy but remain vulnerable to storage access pattern attacks that can link senders and recipients in token transfers. When contracts update recipient balances during transfers, the unique storage keys accessed reveal transaction relationships even when data is encrypted. This paper introduces two novel data structures to address this vulnerability: the Delayed Write Buffer (DWB) and the Bitwise-Trie of Bucketed Entries (BTBE). The DWB delays recipient balance updates by buffering pending transfers and randomly settling entries, breaking the direct correlation between transfer execution and recipient storage access. The BTBE further enhances privacy by grouping addresses into constant-sized buckets, preventing flooding attacks and creating anonymity sets for balance queries. Additionally, we present a private notification system enabling real-time, privacy-preserving push notifications for confidential contracts. Our domain-specific approach leverages the unique characteristics of token transfers -- asymmetric balance updates and tolerance for delayed settlement -- to achieve practical performance with probabilistic anonymity guarantees.

### 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：基于可信执行环境（TEE）的机密智能合约网络（如 Secret Network、Oasis）虽能加密数据，但代币转账时仍存在存储访问模式攻击。合约更新接收方余额时，即使存储键和值已加密，攻击者通过观察合约执行过程中访问了哪些唯一的数据库键，就能推断发送方和接收方之间的关联，从而泄露交易隐私。
- **研究背景**：现有解决方案包括用户主动添加诱饵（使用不便且效果差）和通用型不经意 RAM（ORAM）（开销大、在区块链 gas 限制下不实用）。该论文针对代币转账这一特定领域，利用其非对称余额更新（发送方需立即核验余额，接收方可延迟结算）和对延迟结算的容忍性，设计专门的隐私保护数据结构。

### 2. 方法论：核心思想与关键技术细节
论文提出两种相互配合的链上数据结构和一个隐私通知系统：

- **延迟写入缓冲区（DWB）**
  - **核心思想**：不直接更新接收方的存储余额，而是将待处理转账打包为一个数据条目插入固定大小的缓冲区。缓冲区饱和后，随机选择一个条目“结算”（真正写入接收方存储），从而打破转账执行时间与接收方存储访问之间的时间关联，提供概率匿名性。
  - **关键技术要点**：
    - **数据组织**：每个条目含接收方地址、金额、指向转账事件链表的指针等，整表被整体读取和覆写。
    - **常量时间操作**：所有分支和比较均设计为常量时间，防止侧信道泄露缓冲区状态。
    - **防冲刷（Flush）攻击**：同一接收方重复转账不新增条目，而是累积金额到现有条目中，避免攻击者用大量交易冲刷出特定用户的存储访问。
    - **虚假写入（Phony Writes）**：当发送方或接收方已在缓冲区中时，为避免分支泄露信息，合约总是执行等量的存储写入，对不存在的情况执行无实际效果的虚假写入。
    - **安全性概率**：对于缓冲区容量 k，接收方条目在 n 次后续转账后已被结算（从而暴露存储访问）的概率为  
      \[
      P = 1 - \left( \frac{k-1}{k} \right)^n
      \]  
      例如 k=64 时，达到 99.5% 置信度约需 336 次转账，相当于匿名集大小为 336。

- **分桶条目位字典树（BTBE）**
  - **核心思想**：将用户的最终存储余额分组到恒定大小的“桶”中，通过地址哈希的前导比特构造 Patricia 树，将读写粒度提升到桶级，为余额访问创建匿名集，防止泛洪攻击直接定位到个别地址。
  - **关键技术要点**：
    - 使用合约内部密钥对地址取哈希，外部无法预知分桶。
    - 桶满时分裂叶节点，形成二叉树结构，保证插入效率为 O(log N)。
    - 与 DWB 配合，进一步减少发送方余额访问泄露的风险。

- **隐私通知系统**
  - 为弥补延迟结算带来的体验缺陷，提供加密的实时推送通知。
  - 利用合约内部密钥为每位接收方派生出通知种子，通过 HMAC-SHA256 生成单次使用的通知 ID 并作为交易日志属性发出，只有目标方能解读。
  - 支持三种模式：计数器模式（易用性）、交易哈希模式（抗侧信道）和布隆过滤器模式（高效群发）。
  - 数据使用 ChaCha20-Poly1305 加密，通知载荷常长填充，日志属性数量恒定，以防流量分析。

### 6. 主要结论与发现

- **BTBE 进一步增强隐私**：将余额存储分桶，使存储访问模式膨胀到桶级别，防止单个地址被精准定位。
- **性能可接受**：最坏情况 gas 仅增加 26%，在区块链交易成本中属于可接受权衡。
- **通知系统实用**：推送通知在保障隐私的前提下实现了实时性，多种模式可权衡安全与计算开销。
- **领域特定方法的优势**：相比通用 ORAM，针对代币转账非对称特性的定制化设计可以获得更低的常数开销和 gas 一致性。
