## 摘要
安卓内存取证能够恢复从未触及磁盘的证据：解密的消息、会话凭证以及运行中应用程序的实时内部状态。执行此类恢复的工具依赖于嵌入式在安卓运行时库libart.so中的调试符号，以定位数据结构并解释其布局。在近几个版本中，作为更广泛安全增强工作的一部分，谷歌已从消费级手机搭载的二进制文件中剥离了大部分此类信息，但此前尚无研究衡量在实际检查人员遇到的设备上剥离进展到了何种程度，或者剥离表象之下的内存架构是否仍然类似于取证文献所描述的那样。本文使用从安卓8到安卓15的Pixel工厂镜像中提取的二进制文件，衡量了研究人员使用的未剥离开发版本与真实硬件上搭载的生产版本之间的差距。静态符号从20,495个条目降至零，动态符号下降了大约60%，源文件引用完全消失。安卓15二进制文件中的一个压缩后备段恢复了数千个函数名，但不携带任何结构布局。源代码审查及跨安卓8和15的内存映射对比表明，堆空间、垃圾回收器基础设施和分配位图在结构上保持完整，可见的变化仅限于命名和算术优化。在一台已root且完全剥离的Pixel 7上进行的实时验证确认，运行时入口点仍然可以通过动态符号表定位，并且从版本匹配的开发版本中提取的结构偏移量可解析为生产内存中的有效指针。

## Abstract
Android memory forensics recovers evidence that never touches disk: decrypted messages, session credentials, and the live internal state of a running application. The tools that perform this recovery depend on debug symbols embedded in libart.so, the Android Runtime library, to locate data structures and interpret their layout. Across recent releases Google has stripped most of that information from the binaries that ship on consumer phones as part of a broader security hardening effort, yet no prior work has measured how far the stripping has progressed on the devices examiners actually encounter, or whether the memory architecture beneath the stripped surface still resembles what the forensics literature describes. This paper measures the gap between the unstripped development builds researchers use and the production builds that ship on real hardware, using binaries extracted from Pixel factory images across Android 8 through Android 15. Static symbols fell from 20,495 entries to zero, dynamic symbols dropped by roughly 60 percent, and source file references disappeared entirely. A compressed fallback section in the Android 15 binary restores thousands of function names but carries no structure layouts. Source code review and memory map comparisons across Android 8 and 15 show that the heap spaces, garbage collector infrastructure, and allocation bitmaps remain structurally intact, with visible changes limited to naming and arithmetic optimization. Live validation on a rooted, fully stripped Pixel 7 confirms that the runtime entry point is still locatable through the dynamic symbol table, and that structural offsets pulled from a version-matched development build resolve to valid pointers inside production memory.

- **核心问题**：安卓内存取证工具（如 DroidScraper、Timeliner 等）高度依赖 `libart.so` 中的调试符号（静态符号表、DWARF 信息）来定位运行时数据结构。然而，谷歌近年来对面向消费者的生产版本（user build）进行了激进的安全加固，大规模剥离了这些符号。本论文旨在量化这种剥离的实际程度，并验证在剥离后的生产内存中，安卓运行时（ART）的核心内存管理结构是否依然在布局上与未剥离的开发版本保持一致，从而判断取证工具的跨版本、跨构建类型（develop vs. production）兼容性是否仍然可行。

### 6. 论文的主要结论与发现

- **符号剥离严重但不彻底**：从 Android 8 到 15，生产版 `libart.so` 的静态符号从 20,495 个骤降至零，源文件引用完全消失，动态符号减少了约 60%。仅存的 `.gnu_debugdata` 压缩段只包含函数名，无任何结构布局信息，对取证价值有限。
- **关键入口点幸存**：负责定位 `Runtime` 实例的符号 `_ZN3art7Runtime9instance_E` 是必须的动态链接导出项，因此在所有版本的生产二进制中都能在动态符号表中找到，为取证提供了不可剥离的起点。
- **底层内存架构保持不变**：通过源代码审查和内存映射对比，发现垃圾回收器基础设施、堆空间、分配位图、卡表等核心结构在 Android 8 和 15 之间架构完全保留。可见的变化仅限于命名（如 RosAlloc 主空间改名）和算术优化，不影响数据结构布局。
- **开发构建的偏移量可在生产设备上直接复用（同版本内）**：这是最核心的发现。在 Android 15 的 Pixel 7 上，从版本匹配的未剥离 userdebug 构建中提取的 `heap_`、`thread_list_` 等字段的 DWARF 偏移量，成功解析了生产内存中对应的有效指针，证明了两个构建类型的内部内存布局完全一致。

- **直面实际差距的量化**：首次在真实消费级硬件和官方出厂镜像上对符号剥离进行了精确测量，弥补了先前研究仅依赖开发构建的缺陷。
- **现实可复现的取证工作流**：论文不仅提出问题，还展示了一条可操作的路径：通过获取同版本 CI 构建的 DWARF 信息，作为生产设备的“偏移量地图”。这一流程对数字取证从业人员有直接的指导意义。
- **结合多重证据的严谨论证**：论证并非仅依赖单一维度的对比。它将二进制静态分析、源代码历史追踪、内存映射观察和动态内存注入验证结合起来，形成了坚固的证据闭环，支撑其核心结论——“结构保留”。

### 8. 不足与局限：包括实验覆盖、偏差风险、应用限制等

- **硬件与厂商覆盖极度有限**：所有生产和验证实验均基于谷歌 Pixel 系列手机。三星、小米、一加等厂商的 ART 实现可能包含定制修改，本论文的结论是否适用于这些非 Pixel 设备完全未知，存在显著偏差风险。
- **版本覆盖不连续**：仅分析了 Android 8 和 15 这两个端点，未绘制出符号从部分剥离到完全剥离的详细演化图谱，也未验证中间版本的结构稳定性。
- **验证深度有限**：
  - 仅针对 `system_server` 单个进程进行了实时验证，未覆盖其他类型应用进程。

- **跨版本复用性不成立**：论文明确声明，其“偏移量复用”法只在完全相同的安卓版本内有效。鉴于先前研究已证明核心数据结构偏移量在版本间变动频繁，该方法依然面临巨大的维护负担。
