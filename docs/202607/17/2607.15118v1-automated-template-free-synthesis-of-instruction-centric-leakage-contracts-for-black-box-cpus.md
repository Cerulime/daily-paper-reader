## 摘要
侧信道攻击对现代计算平台构成严重安全威胁，因为它们利用CPU行为中的细微差异来泄漏敏感信息。为了通过微架构侧信道对CPU泄漏的信息进行建模，最近的工作提出了泄漏契约：一种ISA级别的安全抽象，为安全CPU编程提供了基础。不幸的是，由于当前微架构的复杂性，为CPU设计泄漏契约需要大量人工努力，因此现代CPU缺乏专用的泄漏契约。我们提出了一种方法，以最少的人工干预为主要的CPU架构提取以指令为中心的泄漏契约。我们在malcos中实现了该技术，这是首个自动合成黑盒CPU泄漏契约的无模板工具。我们在x86和ARM CPU上评估了malcos，并显示它合成的契约在合成过程中观察到的所有泄漏方面都是精确且可靠的。我们的结果表明，从黑盒CPU学习泄漏契约是可行的。

## Abstract
Side-channel attacks pose a significant security threat for modern computing platforms, because they exploit subtle discrepancies in CPU behaviors to leak sensitive information. To model the information leaked by a CPU via microarchitectural side-channels, recent work proposed leakage contracts: an ISA-level security abstraction that provides the foundations for secure CPU programming. Unfortunately, due to the complexity of current microarchitectures, devising a leakage contract for a CPU requires extensive manual effort and thus modern CPUs lack dedicated leakage contracts.   We present a methodology to extract instruction-centric leakage contracts for major CPU architectures with minimal manual intervention. We implemented this technique in malcos, the first template-free tool that automates the synthesis of leakage contracts for black-box CPUs. We evaluate malcos on x86 and ARM CPUs, and show that the contracts it synthesizes are precise and sound with respect to all leaks observed during synthesis. Our results demonstrate that learning leakage contracts from black-box CPUs is feasible.

## 1. 论文的核心问题与整体含义

- **研究动机**：现代CPU为提升性能引入了大量微架构优化（如寄存器文件压缩、无声存储、乘除法简化运算等），但也带来了侧信道信息泄漏。传统上，开发者依靠“常数时间”模型假设仅控制流和内存访问会泄漏信息，而CPU内部实现常常违反这一假设。  
- **核心问题**：如何精确刻画CPU在ISA层面的侧信道泄漏，以便安全编程者能够规避？  
- **整体含义**：论文提出 **泄漏契约** 是一种ISA级安全抽象，它通过一组条件表达式来指定处理器会在何时泄漏何种信息。然而，手动为商用CPU编写泄漏契约既耗时又需要深度逆向知识，缺乏自动化方法。因此，本文旨在 **以黑盒方式、无模板地自动合成面向指令的泄漏契约**，使现代CPU拥有这类安全规格成为可能。

## 2. 方法论

### 2.1 核心思想：基于反例驱动的契约合成

整个方法（工具名为 `malcos`）将泄漏契约的学习建模为一个 **迭代检查-精炼** 的过程，不依赖处理器RTL设计，也不要求用户提供契约模板。

- **契约语言**：定义了一种称为 `icl` 的语言，每条泄漏条款形如 `ex IF pr`，其中 `ex` 表示泄漏的表达式，`pr` 为触发条件。它可引用寄存器值、操作数类型/访问模式、指令操作码等。
- **合成流程**（参见论文的 Algorithm 1）：
  1. 生成种子测试程序。
  2. 对于每个程序，以空契约或上一个候选契约开始。
  3. 进入循环：
     - **检查阶段（Checker）**：借助关系式泄漏测试工具（x86 用 `Revizor`，ARM 用 `Scam-V`）在真实CPU上执行测试用例，找出违反当前候选契约的 **反例**（即两执行在契约看来相同，但硬件轨迹不同的状态对），同时收集 **正例**（硬件轨迹也相同的状态对）。
     - **精炼阶段（Refiner）**：若发现反例，则将该反例和正例送入基于 `Rosette` 的语法引导合成器，自动生成若干条指令中心的泄漏条款，使得新条款能区分反例且尽可能不区分正例（最大化软约束）。条款通过抽象指令类型被泛化到其他程序。
  4. 当检查器不再发现新反例时，本轮循环结束。
- **后处理**：在多个程序合成得到的候选契约中，执行契约最小化（按精度排序并删除多余条款），以提高契约的精确性，同时保持已发现反例的可区分性。

### 2.2 关键技术细节

- **检查器设计**：使用现有的黑盒关系测试工具，malcos 的框架与检查器解耦。`Revizor` 通过模糊测试生成随机程序与输入，执行并比较硬件轨迹与契约轨迹；`Scam-V` 则结合符号执行和模糊测试。
- **合成约束**：每条新条款必须：
  1. 是面向指令的（谓语中包含指令地址）。
  2. 能区分至少一个反例。
  3. 最大化区分反例且最小化区分正例（软约束）。
- **复位策略**：为避免早期低质量示例导致过度近似，每经过固定数量的程序（参数 `r`）便将候选契约重置为空，再开始新一轮合成，提升最终精度。
- **最小化阶段**：将最终契约中不能提升反例区分能力的条款移除，有效减少无意义的泄漏声明。

整个方法不要求处理器内部设计可见（黑盒），也不依赖用户提供的条款模板，直接从硬件观测中学习。

## 3. 实验设计

### 3.1 数据集、场景与基准

实验在两种架构上展开：x86 和 ARM。

- **RQ1（契约质量）**：以五种预先定义的泄漏模型作为“真值”契约（`ct`、`TagIdx`、`RFC`、`mul`、`SilStore`），模拟这些模型在ISA仿真器上产生的硬件轨迹，从而构造出合成目标。使用 `malcos-x86` 和 `malcos-arm` 分别对每个目标合成契约，并测量精密性（precision）和可靠性（soundness）。验证集为大量随机生成的新程序和状态对。
- **RQ2（最小化影响）**：对比启用/禁用契约最小化阶段后合成契约的质量、条款数量和运行时间。
- **RQ3（真实硬件合成）**：
  - x86：对 4 款 Intel CPU（Skylake、Raptor Lake、Arrow Lake、Lunar Lake）上的 13 个 x86-64 ISA 子集进行合成，使用 `Revizor` 的两种执行器模式（TSC 计时、P+P 缓存集访问）作为硬件观测。
  - ARM：对 Cortex-A72 和 Cortex-A76 上的 30 条 load/store 指令合成契约，硬件观测来自对 L1 缓存状态的监控。
- **RQ4（与模板方法比较）**：定义了 4 个受 RISC-V Ibex 核心启发的泄漏模型（`Ibex-small`、`Ibex-mult-div`、`Ibex-optimizations`、`Ibex-slice`）。将 `malcos` 在与现有模板化工具（`LeaSyn`、`RTL2μPath`、`VeloCT`）相同的受限搜索空间中运行，比较所合成契约的精密性、条款数量和相对执行时间。

### 3.2 对比方法

- **自身消融**：正例数量、复位间隔、合成深度、最小化策略等参数对精度和时间的影响。
- **模板化工具比较**：
  - `LeaSyn`：允许较复杂的模板。
  - `RTL2μPath`：仅支持暴露部分操作数的模板。
  - `VeloCT`：仅支持暴露全部操作数的模板。

## 6. 主要结论与发现

- **可行性**：`malcos` 证明了从黑盒商用 CPU 自动学习精确可靠的指令中心泄漏契约是可行的，无需 RTL 设计或人工模板。
- **合约质量**：在所有 ground truth 模型上，`malcos` 合成的契约均能达到 100% 的可靠性（捕获所有观察到的泄漏）；精密性在大多数情况下达到 100%，仅个别复杂模型因样本质量略有下降（可通过调参改善）。
- **最小化关键**：契约最小化阶段对提升精密性至关重要，移除多余条款可使精密性从 0.13–0.70 提升至接近 1.0。
- **真实硬件结果**：在 x86 和 ARM CPU 上成功捕获了数据缓存地址泄漏、分支泄漏、除法变时性、重复指令计数器泄漏等已知泄漏源，且与 uops.info 等外部延迟测量一致。
- **模板之优势**：对比模板化工具，`malcos` 的无模板特性在泄漏模型较复杂时显著提升了合约的精密性，模板越受限，模板方法的精度劣势越大。

## 7. 优点

- **黑盒与无模板**：不依赖处理器 RTL 设计，适用于所有商用 CPU，且无需手动构造泄漏条款模板，极大降低了使用门槛。
- **自动化程度高**：从测试生成、反例/正例收集、合成、最小化到最终输出契约，几乎不需要人工干预。
- **精确性与可靠性平衡**：通过正例的使用及软约束最大化，合成的契约在捕获新泄漏的同时尽量避免了过度近似。
- **架构抽象**：检查器可插拔，目前支持 `Revizor` 和 `Scam-V`，易于扩展至其他观测模型或架构。
- **实测验证**：在真实硬件（多款 Intel 和 ARM CPU）上进行了大规模实验，并对比了公认的延迟数据库，证明了合成的合理性。
- **参数可调**：复位策略、正例数量、合成深度等均可调整以适应不同需求，论文给出了详细的消融和推荐值。

## 8. 不足与局限

- **仅限指令中心契约**：`malcos` 目前只能合成单指令触发的泄漏条款，无法捕捉跨越指令的状态性泄漏（如计算重用、操作数打包等）。
- **用户模式局限**：使用的检查器不支持特权指令，因此无法发现内核级泄漏（例如页表访问）。
- **假阴性风险**：合成的合同可靠性是“有界的”，只能保证捕获测试过程中观察到的泄漏，未被测试用例触及的微架构泄漏仍然可能遗漏。
- **假阳性可能**：尽管有最小化，但契约仍可能产生过度近似，特别是当泄漏本质难以从有限的测试输入精确推断时（如仅泄漏寄存器哈希值的情况）。
- **执行开销**：合成过程需要大量模糊测试执行，对 x86 每款 CPU 仍需数十小时，ARM 上单板运行也需要数十小时，虽可并行但总体计算代价不低。
- **检查器依赖**：合成质量高度依赖于底层检查器发现泄漏的能力及其生成正/反例的质量。如果检查器未能发现某些泄漏或提供了误导性样本，最终契约的完整性和精确性将受损。
- **ISA 覆盖不全**：由于 `Revizor` 的限制，x86 实验未覆盖所有扩展指令（如 AVX、x87），间接跳转等部分控制流指令也未支持。
