## 摘要
WebGPU允许普通网页通过经过验证的编程模型运行GPU工作负载。验证保护了内存安全，但共享的浏览器、驱动、操作系统和GPU状态仍可能暴露与隐私相关的信号。我们提出了WGPULens，一个用于在受控场景、浏览器原生共存、参与者实地研究、公开页面加载以及缓解策略中测量这些信号的框架。我们的框架将测量分开：受控场景支持泄漏、边界和缓解声明；参与者运行支持部署、兼容性和指纹识别性；而Tranco爬虫测量了现实网页中的WebGPU暴露情况。我们的受控结果识别出持久的管线编译状态是最明显的表面。冷/热管线探测揭示了在选定来源、配置文件和浏览器放置下先前的编译状态。受控的浏览器/原生实验还表明，在标记的工作负载下，可以从浏览器可见的可观测量中推断出原生GPU活动。其他资源探测提供了较弱的正面结果和阴性对照。参与者实地研究显示，活跃的WebGPU行为在样本中高度独特，其中确定性组件在运行中稳定，但跨重复访问的精确稳定性较低。页面加载爬虫发现WebGPU的使用主要集中在适配器探测和静态支持代码上，未观察到页面加载时的着色器、管线、队列、查询或映射活动。缓解试点识别出源级密钥分离作为评估管线缓存分区的代理。总体而言，WGPULens表明WebGPU隐私分析必须是表面特定的：浏览器需要测量哪些GPU状态跨越了哪些边界，哪些浏览器可见信号揭示了这一点，以及相应的缓解措施成本如何。

## Abstract
WebGPU lets ordinary web pages run GPU workloads through a validated programming model. Validation protects memory safety, but shared browser, driver, OS, and GPU state can still expose privacy-relevant signals. We present WGPULens, a framework for measuring those signals across controlled scenarios, browser-native co-residency, a participant field study, public page loads, and mitigation policies. Our framework separates measurements: controlled scenarios support leakage, boundary, and mitigation claims; participant runs support deployment, compatibility, and fingerprintability; and a Tranco crawl measures WebGPU exposure in real-world pages.   Our controlled results identify persistent pipeline compilation state as the clearest surface. Cold/warm pipeline probes reveal prior compilation state across selected origin, profile, and browser placements. Controlled browser/native experiments also show native GPU activity can be inferred from browser-visible observables under labeled workloads. Other resource probes provide weaker positive results and negative controls.   The participant field study shows active WebGPU behavior is highly distinctive within the sample, with deterministic components stable within runs and lower exact stability across repeated visits. A page-load crawl finds WebGPU use mainly as adapter probing and static support code, with no observed page-load shader, pipeline, queue, query, or map activity. Mitigation pilots identify source-level key separation as a proxy for evaluating pipeline-cache partitioning. Overall, WGPULens shows that WebGPU privacy analysis must be surface-specific: browsers need to measure which GPU state crosses which boundary, which browser-visible signals reveal it, and what the corresponding mitigations cost.

### 1. 论文的核心问题与整体含义

*   **研究动机：** WebGPU 的标准化使网页能够直接利用 GPU 进行可编程计算，极大地扩展了浏览器端的计算能力。然而，虽然 WebGPU 的验证层确保了内存安全，但它依赖于浏览器、驱动、操作系统和 GPU 硬件多层共享状态。
*   **核心问题：** 这种共享状态可能形成**隐私侧信道**，攻击者（普通网页）通过合法的 WebGPU API 便可观察到这些状态，从而推断出跨源、跨配置文件的用户历史行为，甚至同一设备上原生 GPU 应用的活动。
*   **整体含义：** 论文旨在系统性地**测量并量化 WebGPU 引入的隐私风险**，明确指出浏览器中哪些 GPU 状态跨越了哪个边界、能被哪些可见信号揭示，以及采取缓解措施需要付出何种代价。研究强调，WebGPU 隐私分析必须是“表面特定”的，而非一刀切。

### 2. 论文提出的方法论

*   **核心思想：** 提出一个名为 **WGPULENS** 的测量框架，将测量实验分为三种互补的路径，分别回答不同维度的隐私问题：
    *   **受控场景（Controlled Scenarios）：** 固定边界、受害者负载和标签，用于支持**泄漏、边界和缓解声明**。
    *   **参与者实地研究（Participant Field Study）：** 在真实、多样的浏览器和硬件环境中收集数据，用于评估**部署兼容性、异构性和样本级指纹识别性**。
    *   **页面加载爬虫（Page-load Crawl）：** 对公开网站进行大规模爬取，测量**现实中网页对 WebGPU 的暴露情况**。

*   **关键技术细节与流程：**
    *   **探针生成：** 所有探针均为**合法的 JavaScript 和 WGSL 代码**，严格遵守 WebGPU 验证边界，避免越界访问、未初始化读取等非法操作。探针覆盖三大类表面：
        *   **静态暴露：** 适配器元数据、特性、图形签名等。
        *   **活跃争用：** 内存带宽、纹理、原子操作、屏障、队列进度等。
        *   **持久状态：** 着色器/管线编译状态（冷/热路径）。

### 6. 论文的主要结论与发现

*   **管线编译状态是最清晰的隐私表面：** 浏览器/驱动栈中的管线缓存复用策略成为主要的持久性状态泄漏源。攻击者通过探测创建管线时的冷/热路径差异（基于高精度时间测量），可以有效推断用户先前是否存在过相关的 GPU 活动。该信号在跨源、HTTPS iframe、跨 Chromium 系浏览器等边界下依然为正（AUROC 值高），但在降低时间精度或仅做顺序观察时显著减弱。
*   **原生 GPU 活动可被推断：** 在受控的浏览器/原生共驻条件下，一个 WebGPU 网页能够**推断出原生 GPU 应用（OpenCL/Metal）的活动甚至工作阶段**，这主要通过粗粒度的帧层次可观测信号实现。
*   **WebGPU 行为具有高度指纹区分性：** 参与者现场数据显示，结合了输出哈希、稳定性标记和时序摘要的“活跃 WebGPU 签名”在样本中能达到 **100% 的唯一性**（1,095 个去重记录产生 1,095 个匿名集）。相比而言，仅依靠静态适配器信息则仍有大量碰撞。
*   **现实世界页面使用程度低：** 顶级网站爬虫发现，页面加载期间的 WebGPU 使用几乎仅限于**适配器探测和静态支持代码**，未观察到任何着色器编译、管线创建、队列提交或内存映射等实际 GPU 工作。
*   **缓解措施存在权衡：** 以**源级密钥分离**作为管线缓存分区策略的代理，可以在显著降低 AUROC（泄漏量）的同时，仅引入数毫秒的编译延迟成本，指明了一个明确的浏览器端缓解方向。

### 7. 优点

*   **框架设计精巧且系统：** 提出的 WGPULENS 框架通过三条独立路径（受控、参与者、爬虫）有机地解耦了不同的隐私主张，避免了论证混淆，设计严谨。
*   **表面特异性分析方法：** 研究并未笼统地看待 WebGPU 隐私，而是将攻击表面细分为静态、争用和持久状态，并指出了**持久管线状态是最主要的风险点**，为后续的防御提供了精确的靶点。
*   **严格的控制与对照实验：** 大量使用阴性对照（如跨标签页应无泄漏）、标签置换检验和多日重复实验，确保了正结果的可靠性，排除了测量噪声或过拟合。
*   **结合了“实地”与“爬虫”视角：** 不仅停留在实验室内，还通过参与者和爬虫数据，评估了该隐私风险在真实设备上被放大的程度（高指纹识别性）以及在真实网页中当前利用的现状（低使用度），评估维度全面。
*   **关注缓解成本：** 在讨论防御措施时，明确将泄漏减少量与**用户可感知的成本（编译延迟）** 结合起来评估，提供了可行的策略建议，而非仅指出问题。

### 8. 不足与局限

*   **根因定位不彻底：** 虽然有对照，但信号的根本原因可能散布在浏览器层（如 Dawn/wgpu）、驱动缓存、操作系统文件缓存等多层，论文承认其测量结果仅归因于“浏览器/GPU 栈”，无法精准定位到某一具体层。
*   **受害者工作负载为合成：** 受控实验中推断原生 GPU 活动或网页工作负载的受害者均为**人工设计的合成负载**，这与现实世界中复杂、多变的应用程序运行状况存在差距，外部有效性受限。
*   **参与者数据的覆盖与偏差：** 参与者研究是“选择加入”的，其样本可能存在**志愿者偏差**，且在移动端（Android/iOS）、Safari 等特定配置上的覆盖可能不如 Chrome/Windows 充分。此外，跨会话、浏览器重启、系统更新后的长期指纹稳定性尚未验证。
*   **缓解措施的实验级验证：** 源级密钥分离只是一个**模拟/代理**，不完全是生产环境中浏览器引擎级的缓存键修改。真正的浏览器侧实现可能面临更复杂的兼容性和性能代价。
*   **静态防护不足：** 单纯靠降低时间分辨率或调整调度器策略等动态防护，对活性输出指纹（如计算/渲染结果的哈希）这类确定性信息的防护效果有限，需要专门的读回扰动或隐私模式。
