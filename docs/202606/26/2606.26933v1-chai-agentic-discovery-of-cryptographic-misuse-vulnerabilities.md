## 摘要
AI辅助的漏洞发现已在内存安全等漏洞类别中证明有效，此类漏洞可通过插桩确认内存违规并高效过滤误报。然而，许多危险的漏洞类别，例如密码误用，却缺乏类似的插桩手段。本文提出Chai，一个基于AI的系统，通过自然发生的信号来发现和确认密码误用漏洞。为此，Chai重新思考了经典的差分测试技术，利用AI以：1）提高对库中真实安全问题检测的精确度；2）将常被忽视的差异重新用作下游应用中切实漏洞的线索。在此过程中，Chai颠覆了当前AI漏洞发现的范式：它不再针对单个代码库审计众多缺陷，而是在库层级编目缺陷，并沿密码依赖图传播这些缺陷，从而实现复合效率增益。我们在X.509、JWT和SAML库上评估了Chai。Chai在一个驱动数十亿设备的SSL库中发现了一个此前未知的严重漏洞，在一个主流浏览器背后的库以及多个主流Linux发行版的库中发现了安全缺陷。这些技术共揭示出超过100个漏洞。

## Abstract
AI-assisted vulnerability discovery has proven effective for bug classes like memory safety, where instrumentation confirms memory violations and efficiently filters false positives. Many dangerous vulnerability classes, such as cryptographic misuse, however, lack any comparable instrumentation. In this work, we present Chai, an AI-based system that discovers and validates cryptographic misuse vulnerabilities through naturally occurring signals. To achieve this, Chai rethinks the classical technique of differential testing by leveraging AI to 1) improve precision for detecting real security issues in libraries, and 2) repurpose commonly overlooked discrepancies as leads for tangible vulnerabilities in downstream applications. In doing so, Chai inverts the prevailing paradigm of AI vulnerability discovery: instead of auditing one codebase for many flaws, it catalogs flaws at the library level and propagates them across a cryptographic dependency graph, delivering compounding efficiency gains. We evaluate Chai across X.509, JWT, and SAML libraries. Chai discovered a previously unknown critical vulnerability in an SSL library that powers billions of devices, along with security bugs in one library behind a major web browser and another in major Linux distributions. In total, these techniques surfaced over 100 vulnerabilities.

### 2. 方法论：核心思想与关键技术
Chai 分为两大阶段：**增强式差分测试** 与 **差异追踪**，两者通过“差异”这一信号衔接。

#### 2.1 第一阶段：增强式差分测试
- **输入构造**：从合法种子出发，由变异智能体生成“变异程序”（操作序列），再经确定性构建器产出跨库统一的协议消息（X.509 证书、JWT 或 SAML 断言）。
- **放大测试**：单条输入同时提交给所有已适配的库（X.509 13 个、JWT 23 个、SAML 11 个），以各库返回值（accept/reject）是否一致作为测试先知，任一不一致即产生一个“差异”。
- **自适应搜索与资源分配**：
  - 变异由检索增强生成（RAG）驱动，从 CVE 历史中提取攻击家族，形成“变异组”（mutation groups）。
  - 采用多臂老虎机算法（UCB1）为并行的变异组分配租约，只根据唯一的新差异给予奖励，避免重复搜索已探索区域。
- **差异分析与分类**：对差异进行重放、最小化，然后分为四类：
  1. 漏洞（无条件可被攻击者利用）
  2. 安全缺陷（违反规范但需额外条件）
  3. 模糊性（规范允许的合法差异，但可能被下游误解）
  4. 非安全差异
  其中，“模糊性”进入第二阶段。

#### 2.2 第二阶段：差异追踪
- **下游枚举**：从 OpenSSF 关键性评分最高的 50K 项目中构建依赖图（涵盖 17 个生态），找出依赖了表现出模糊行为的库的应用。
- **针对性审计**：只检查应用是否依赖库未提供的安全属性（如信任令牌内嵌密钥），而非开放式搜索，大幅降低任务难度。
- **PoC 生成与人工确认**：在沙箱 VM 中生成端到端概念验证，由评判智能体迭代精化，最终由安全专家审核并生成报告。所有报告均经人工复核后才公开。

**算法流程（文字描述）**：
① 初始化种子与变异组池，每个组对应（信任边界，操作）对。
② 并行搜索中，每次迭代：检索智能体获取相关 CVE 模式 → 变异智能体产生操作序列 → 构建器生成输入 → 所有库执行 → 记录差异向量。
③ 对差异去重与最小化 → 分类器判断类别。
④ 对环境中的模糊差异，查询依赖图 → 审计智能体对每个候选应用进行“是否依赖缺失保障”的闭环检查 → PoC 管道生成利用脚本。
⑤ 租约管理器根据 UCB1 分数更新各组的分配，搜索持续直至预算耗尽。

### 7. 优点
- **范式创新**：通过库级差异反向追踪至应用，将一次审计的成果扩散至整个生态，复合效率增益显著。
- **AI 与符号验证结合**：用差分测试提供可重复的确定性信号，解决了 AI 漏洞发现中的可信度问题。
- **免手工语法**：智能体自适应搜索，无需为每个新协议编写语法，支持多协议泛化。
- **精细的资源调度**：UCB1 驱动的租约机制在并行搜索中平衡探索与利用，避免大量重复劳动。
- **实用化流程**：包含 PoC 生成、报告起草与人工审核，确保产出的严重漏洞可被及时修复，且不给维护者增加假阳性负担。
- **充分的实验验证**：在多个真实协议和大量库上评估，与多个基线及行业级智能体真实盲测结果对比，证据扎实。

### 8. 不足与局限
- **依赖多实现**：若协议仅有单一实现，差分测试无法工作，Chai 不适用。
- **库适配质量影响**：Harness 必须准确调用库接口，若某库不完全遵循规范，会引入噪声，降低差异的信噪比。
- **依赖图覆盖**：仅基于 OpenSSF 关键性前 50K 项目，可能遗漏小众但重要的下游应用。
- **漏洞类型受限**：仅限于通过黑盒输入就能触发的协议行为分歧，无法发现库内部的逻辑缺陷（如密钥生成错误）或数学级别弱点。
- **自动审计尚不完备**：尽管 PoC 流水线自动化程度高，最终仍需人工核实和报告修改；目前不能全自动提交漏洞。
- **实验范围**：仅测试了三个密码协议，在其他认证/授权协议上的有效性待进一步验证。
- **成本虽低，但仍受限于推理服务**：持续大规模运行仍需实时接入 LLM API，可能需要更多资源以覆盖更深层状态。
