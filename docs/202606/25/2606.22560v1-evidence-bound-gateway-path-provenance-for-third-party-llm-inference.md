## 摘要
第三方大语言模型网关已成为应用程序与外部大语言模型提供商之间的关键基础设施层。传统网关不仅仅转发流量：它们决定调用哪个提供商和模型，是否发生回退，交付哪个流，以及应计费的使用记录。由于这些决策和记录是在运营商控制的服务内部生成的，客户端无法独立区分诚实的中间服务与路由替换、隐藏回退、流操纵或伪造溯源。我们提出了一种基于证据的大语言模型网关架构，将运营商控制平面与经过认证的执行平面分离。在网关内部，经过度量的认证网关运行时（AGR）是唯一允许解密请求、执行路径策略、构建上游调用和签名证据的组件。客户端在将请求加密到绑定于AGR度量值的密钥之前，验证签名的发布元数据和新鲜认证。AGR强制执行请求范围的路由、回退和端点约束，调用被许可的提供商，返回加密响应流，并签名证据，将策略、所选路由、端点身份、流承诺和完成元数据绑定到经过认证的运行时。在AWS Nitro Enclaves上的初步Rust原型显示出适中的机制开销，并能通过故障关闭检测到在认证运行时之外对策略、路由、端点和流证据的篡改。

## Abstract
Third-party LLM gateways have become a critical infrastructure layer between applications and external LLM providers. Conventional gateways do more than forward traffic: they decide which provider and model are called, whether fallback occurred, which stream is delivered, and what usage record should be billed. Because these decisions and records are authored inside the operator-controlled service, clients cannot independently distinguish honest mediation from route substitution, hidden fallback, stream manipulation, or forged provenance.   We present an evidence-bound LLM gateway architecture that separates the operator control plane from an attested execution plane. Within the gateway, a measured Attested Gateway Runtime (AGR) is the only component allowed to decrypt requests, enforce path policy, construct upstream calls, and sign evidence. Clients verify signed release metadata and fresh attestation before encrypting requests to keys bound to the AGR measurement. AGR enforces request-scoped routing, fallback, and endpoint constraints, invokes admitted providers, returns encrypted response streams, and signs evidence binding the policy, selected route, endpoint identity, stream commitments, and completion metadata to the attested runtime. An initial Rust prototype on AWS Nitro Enclaves shows modest mechanism overhead and fail-closed detection of policy, routing, endpoint, and stream-evidence tampering outside the attested runtime.

### 1. 论文的核心问题与整体含义

*   **研究背景**：第三方大语言模型网关已成为应用与多个外部LLM提供商之间的关键中间件。传统网关在运营商控制的服务内部完成路由选择、故障回退、流式传输、使用记录与计费，使得客户端无法独立验证网关的实际执行路径。
*   **核心问题**：客户端接收到的响应和网关自行生成的日志，无法区分诚实的中间服务与被篡改的路径（如模型替换、隐藏回退、端点重写、流操纵或溯源伪造）。尤其在代理场景下，流内容可能驱动工具调用、代码修改等副作用，因此网关已成为LLM推理供应链的一部分，其路径真实性至关重要。
*   **整体含义**：论文将第三方LLM推理视为一个网关路径溯源问题，目标是让客户端通过密码学证据自主验证网关运行时、策略执行、路由选择、回退决策、端点到流完整性，从而打破对运营商自我记录的单方面信任。

### 2. 论文提出的方法论

*   **核心思想**：将网关拆分为运营商控制的业务平面与经过远程证明的执行平面。所有路径关键决策均在可度量证明的“经认证网关运行时（AGR）”内完成，并由AGR生成不可伪造的签名证据，客户端通过验证该证据来确保路径可信。
*   **关键组件与流程**：
    *   **发布注册表**：发布权威对AGR镜像、度量值、策略等签名，客户端固定公钥验证签名、新鲜度与策略一致性。
    *   **新鲜认证与密钥绑定**：客户端采样新鲜nonce，请求AGR的硬件证明（如AWS Nitro证明文档），验证度量值、策略哈希、加密密钥、证据签名密钥等的绑定。
    *   **策略承载推理合约（IC）**：客户端在每个加密请求中携带一个缩小允许范围的策略对象。AGR执行时计算有效策略`P_eff = P ∩ IC`，进一步约束路由、回退与端点。
    *   **加密通信**：客户端通过Noise NK或HPKE建立会话，加密请求体，AGR解密后依据`P_eff`实施路由与回退决策，并对上游调用进行TLS验证。响应流被加密返回。
    *   **推理证据链（IEC）**：AGR签名一个结构化证据对象，绑定会话上下文、策略哈希、合同哈希、所选路由、回退记录、端点观察（`E_obs`）、加密流承诺（`StreamEv`）、完成元数据等。其中`StreamEv`采用滚动哈希`S_i = H(S_i-1, sid, reqid, stream, i, C_i, metadat_i)`保证流完整性。可选地使用HMAC生成请求/响应承诺供客户端验证。
    *   **验证**：客户端接收IEC后，重新计算`StreamEv`并比对签名，验证所有绑定关系。网关自身也可验签但只能看到密文和元数据。这实现了“证据绑定”的端到端路径溯源。

### 6. 论文的主要结论与发现

*   **可验证路径溯源**：通过引入AGR和签名证据链，客户端无需信任网关业务平面的自述记录，即可验证请求是否经过合法运行时、遵守了指定的路由/回退/端点策略，以及流是否被篡改。
*   **机制开销可控**：在本地mock和Nitro Enclave原型中，证据绑定带来的延迟和吞吐量开销保持在1.1倍以内，证据操作（如IEC验证）本身仅需数十微秒，Nitro证明为一次性会话开销（约50ms）。
*   **故障关闭检测有效**：故障关闭验证证明，任何在AGR外部对证明、注册表、密钥、路由、流或证据的篡改均会导致客户端验证失败，拒绝接受结果。
*   **信任范围收窄**：设计没有消除对发布密钥、TEE实现等底层信任，但成功将网关路径上的信任集中到经过证明的硬件安全区域，而不覆盖上游LLM提供商的内部行为。

### 7. 优点

*   **架构清晰**：分离业务平面与证明平面，将安全关键逻辑收拢至AGR，思维清晰。
*   **证据链设计精巧**：IEC与`StreamEv`提供了隐私保护（承诺而非明文）的路径绑定证据，支持流式完整性验证。
*   **端到端流程完整**：涵盖注册表-认证-合约-加密-签名-验证的全流程，且客户端仅需验证签名证据，不接触业务日志。
*   **实验验证多层次**：不仅有微基准、并发矩阵，还包含Nitro环境验证、故障注入以及真实端点兼容性，设计较为完整。
*   **安全性论证严谨**：给出了引理和安全性定理，明确了假设和泄漏，对非目标也做了清晰说明。

### 8. 不足与局限

*   **非生产级规模验证**：实验仅基于小规模确定性mock和少数真实请求，缺乏大规模异构负载下生产环境的性能与稳定性数据。
*   **模型执行不验证**：AGR仅保证网关路径，不保证上游LLM提供商内部确实使用了声明模型或返回安全内容。这需要提供商侧的可验证计算证据。
*   **元数据泄漏未解决**：流量时序、数据包大小、请求频率等信息仍对网关可见，论文明确将其列为非目标。
*   **现有TEE局限**：依赖TEE平台的信任根、远程证明链、发布签名过程以及客户端实现无漏洞，面临侧信道等传统TEE风险，论文对此有明确表述。
*   **覆盖范围受限**：原型未实现多模态、嵌入等API，错误封装也不完整；实验仅在AWS Nitro Enclaves上进行，未展示跨TEE平台的通用性。
*   **依赖离线验证**：故障关闭检测需要客户端正确实现验证逻辑，若客户端实现有误或被绕过，则安全属性丧失。
