## 摘要
GPU机密计算（GPU-CC）现已保持GPU本地性能：在NVIDIA B300上，BF16矩阵乘法运行速度达到非机密性能的0.998倍。然而，在Intel TDX加GPU-CC下的LLM服务仍会损失13-27%的吞吐量，KV缓存恢复延迟可能翻倍以上。本文在两个Blackwell平台（RTX Pro 6000和B300 HGX）上研究了这一差距，并确定了其主要原因：机密VM与GPU之间的桥接，而非GPU计算。我们发现GPU-CC将主机/设备间的数据移动变为一个串行化、高启动开销的通道。安全拷贝在同一上下文内无法获得CUDA流的并发性，异步传输在运行时边界处阻塞，且小量数据传输需支付固定开销。这违背了现代推理运行时的假设，即DMA操作应廉价、并发且异步。在vLLM密集解码中，差距集中在慢约44倍的小型分配与拷贝操作；定向补丁排除了其他解释。一个调度标志可恢复57%的差距，而在合格的高并发运行中，工作者线程排空可恢复高达92%。同一桥接模型解释了+131%的KV恢复损失和34倍的模型加载减速。Blackwell还改变了机密租户单元。我们在B300上验证了机密多GPU NVSwitch租户，包括CVM内510 GB/s的NVLink P2P及并发隔离租户，并指出了生产级机密AI平台剩余的结构证明差距。

## Abstract
GPU Confidential Computing (GPU-CC) now preserves GPU-local performance: on NVIDIA B300, BF16 matmul runs at 0.998x of non-confidential performance. Yet LLM serving under Intel TDX plus GPU-CC still loses 13-27% of throughput, and KV-cache restore latency can more than double. This paper studies that gap on two Blackwell platforms, RTX Pro 6000 and B300 HGX, and identifies its dominant cause: the confidential VM-GPU bridge, not GPU compute.   We find that GPU-CC turns host/device movement into a serialized, high-setup-cost channel. Secure copies do not gain CUDA-stream concurrency within a context, asynchronous transfers block at the runtime boundary, and small crossings pay a fixed toll. This violates the assumptions of modern inference runtimes, where DMA is expected to be cheap, concurrent, and asynchronous. In vLLM dense decode, the gap closes around 44x-slower small alloc-and-copy operations; targeted patches reject alternative explanations. A scheduling flag recovers 57% of the gap, while a worker-thread drain recovers up to 92% in qualified high-concurrency runs. The same bridge model explains a +131% KV-restore penalty and a 34x model-load slowdown.   Blackwell also changes the confidential tenancy unit. We qualify confidential multi-GPU NVSwitch tenants on B300, including 510 GB/s NVLink P2P inside a CVM and concurrent isolated tenants, and identify the remaining fabric-attestation gap for production confidential AI platforms.

### 1. 论文的核心问题与整体含义

*   **研究动机**：尽管 NVIDIA Blackwell GPU 的机密计算（GPU-CC）已几乎不损失本地计算性能（BF16 矩阵乘达 0.998x 于非机密性能），但在 Intel TDX 保护下的机密虚拟机（CVM）中运行 LLM 服务时，端到端吞吐量仍下降 13-27%，KV缓存恢复延迟翻倍以上，模型加载时间可达 34 倍。这一差距随硬件代际更替而放大（B300 上的相对损失远大于 Hopper），表明瓶颈不在 GPU 计算而在数据传输路径。
*   **核心问题**：研究在 Blackwell 平台上 GPU 机密计算下，LLM 服务性能损失的根源，并提出可恢复方法。论文目标是定位性能税发生的精确位置，并证明其并非 GPU 计算，而是 CVM 与 GPU 之间的“桥接”通道。
*   **整体含义**：GPU 机密计算的成本不应被看作一个固定数值，而应理解为一种“契约变更”：GPU 本地计算几乎免费，但主机与设备间的数据移动变为串行化、高启动开销的通道，且异步性被静默撤销。现代推理运行时对此缺乏适配，导致了巨大的性能损失，但通过软件层面的针对性优化，大部分损失可以被挽回。

### 2. 论文提出的方法论

论文的核心方法并非提出一个全新的系统，而是通过**微观基准测试与系统性剖析**，建立了一个**可预测的性能模型**（“串行化桥接”模型），并据此设计了一系列**软件层面的恢复策略**。

*   **核心思想：串行化桥接模型**
    论文通过剖析发现，在机密模式下，CVM-GPU 桥接呈现三个关键特性：
    1.  **无上下文内并发**：同一 CUDA 上下文内的所有安全拷贝共享少量固定通道，即使 16 个流同时执行小拷贝也几乎没有并行度；额外带宽只有通过增加 CUDA 上下文数量才能获得。
    2.  **异步性被静默撤销**：`non_blocking` 拷贝会阻塞调用线程，因为驱动程序在回弹缓冲设置完成前无法返回。
    3.  **每次跨越有固定通行费**：因 TDX 私有内存无法被 GPU DMA 直接访问，每次主机到设备的拷贝都需要经过加密暂存路径，产生约 330 微秒的固定设置开销，对小数据量传输惩罚极大。

*   **关键技术细节与恢复策略**
    基于上述模型，论文按“成本层级”设计了从简单到复杂的恢复措施：
    *   **调度策略修改**：
        *   **标志级修复**：禁用 vLLM 的默认异步调度（`--no-async-scheduling`）。此标志将工作模式改为顺序的“抽样-少量传输-排空”模式，避免了因试图重叠传输而造成的通道串行化和仲裁开销。
        *   **补丁级修复**：将阻塞性输出排空（drain）移至工作者线程（worker-thread drain）。由于 CUDA 调用阻塞时会释放 GIL，将排空操作放在 Python 工作线程中可以恢复主机端的流水线能力，即 CPU 准备下一步计算时，工作线程在并行等待 DMA 完成。
    *   **数据搬运工程**：
        *   **上下文池化加载器**：利用“带宽存在于上下文数量”的发现，将模型权重分片并行加载到多个工作者 CUDA 上下文中，再通过 GPU 内 Peer 拷贝重组。同时，测量了安全上下文昂贵的生命周期（创建约 5.2 秒，销毁约 3.9 秒），并采用预热的持久化上下文池来分摊开销，最后将上下文销毁异步化以完全隐藏成本。
        *   **重用感知的 KV 缓存卸载**：鉴于桥的昂贵，将默认的“全量溢出”策略改为仅卸载被多次访问的 KV 块，极大减少了跨桥数据量。

### 6. 论文的主要结论与发现

1.  **性能瓶颈定位**：GPU 机密计算下 LLM 服务性能损失的**主导原因是 CVM-GPU 桥接的串行化**，而非 GPU 计算。GPU 本地计算（矩阵乘等）性能几乎无损失（0.998x-1.001x）。
2.  **桥接特性**：机密模式将主机-设备数据传输变成：a) 每上下文内并发缺失的串行通道；b) 异步传输静默变为同步阻塞；c) 每有一次跨桥操作需支付~330 微秒的固定高额开销。这些特性在现代多流、重叠传输的推理运行时中造成了**策略倒置**（原本有益的异步重叠变成了纯粹的损失）。
3.  **恢复策略与效果**：
    *   **一个调度标志**（禁用异步）可恢复密集解码 57% 的吞吐差距，并几乎完全消除 KV 恢复延迟惩罚。
    *   **工作者线程排空**机制可在高并发下恢复高达 92% 的吞吐差距（有条件声明）。
    *   **上下文池化加载器**将 GPT-OSS-120B 模型加载时间从 287 秒缩短到 8.4 秒（34 倍提速），证明了数据搬运工程的有效性。
4.  **平台能力**：在 B300 上首次公开验证了**机密多 GPU NVSwitch 结构分区租户**。可以在 CVM 内实现 510 GB/s 的 NVLink P2P，支持并发隔离租户，但也指出了**结构证明的缺失**：结构管理器、路由表等控制平面组件仍不在租户的可验证证据中，存在信任缺口。

### 7. 优点

*   **机制性分析**：不仅报告现象，更深入建立了“串行化桥”的因果模型，从底层微基准到上层应用完整串联，解释了各种负载下的性能损失差异。
*   **实用且可操作**：给出的解决方案层次分明，从简单的命令行标志到中等复杂度的代码补丁，再到架构级的原则，可直接指导生产部署和框架设计。
*   **跨平台验证**：在两个硬件架构迥异的 Blackwell 平台（Pro 6000 和 B300）上验证了桥接定律和大多数修复的一致性，增强了结论的通用性，并通过 Hopper 的边界实验确认该机制并非新出，只是效应放大。

### 8. 不足与局限

*   **实验复现性与统计力度**：最亮眼的 B300 恢复数据（92%）因硬件访问时间有限，仅为单次测量，未能独立复现，其普适性存疑。核心结论虽不依赖此数据，但一定程度上削弱了论述的完美程度。
*   **优化范围未闭环**：论文明确指出，要完全消除剩余的性能差距，需要多步 CUDA Graphs 或底层架构支持（如 TDISP/TEE-IO 可信设备路径）。软件层面的优化已到边界，彻底的硬件透明化仍属未来。
*   **结构验证不完整**：对 B300 的多 GPU 结构验证，8-GPU 等工作负载因硬件/固件 bug 未完成，NCCL 集合通信和 CC 开关下 NVLink 的对比也因故空缺，留下了重要的能力缺口。
*   **安全性信任缺口**：文中明确指出了结构证明缺口，即控制平面（如结构管理器）仍为宿主机可信，这在安全严谨的生产环境是一个显著的残余风险，论文提供了方向但没有给出解决方案。
*   **通用性边界**：所有优化和结论都紧密围绕 NVIDIA Blackwell GPU 与 Intel TDX 的组合。虽然通过桥接定律解释了核心机制，但这些修补措施（如修改 vLLM 特定调度逻辑）在其他 GPU 架构或 TEE 方案中的可移植性未被探讨。
