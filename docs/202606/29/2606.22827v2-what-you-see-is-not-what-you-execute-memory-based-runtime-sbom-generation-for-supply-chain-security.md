## 摘要
现代软件开发高度依赖来自公共仓库的第三方组件，扩大了软件供应链的攻击面。为应对这些日益增长的风险，联邦政府倡议推动软件物料清单（SBOM）作为一种标准化机制，通过描述软件组件、依赖关系及其关联来提升透明度。然而，基于元数据或文件系统工件构建的SBOM无法捕获运行时加载和执行的组件，尤其是在Python等动态生态系统中。此外，通过插桩生成运行时SBOM需要提前部署监控，并且系统在执行期间保持可观测。在生产环境和事件响应场景中，这些条件难以满足。相比之下，易失性内存提供了恢复运行中应用程序实际运行时状态的可靠来源，无需预先插桩。因此，本文提出MEM-SBOM，这是第一个直接从Python应用程序运行时状态生成SBOM的内存取证框架。它从解释器的内部结构恢复模块，解析包版本，并分析字节码以构建依赖图和识别易受攻击的函数。我们将MEM-SBOM实现为一套Volatility 3插件，并在51个真实世界的Python应用程序上进行了评估。它实现了100%的提取准确率，识别出Streamlit是唯一调用tornado依赖中易受攻击例程的应用程序，并恢复了现有SBOM工具遗漏的所有运行时包，提供了更准确的依赖图和更好的漏洞评估。这些能力使MEM-SBOM通过提供系统上实际执行内容的取证可靠的运行时视图，成为软件供应链安全和事件响应的实用基础。

## Abstract
Modern software development relies heavily on third-party components from public repositories, expanding the software supply chain attack surface. In response to these growing risks, federal initiatives have advanced the Software Bill of Materials (SBOM) as a standardized mechanism for improving transparency by describing software components, dependencies, and their relationships. However, SBOMs built from metadata or filesystem artifacts fail to capture the components loaded and executed at runtime, especially in dynamic ecosystems such as Python. Moreover, generating runtime SBOMs through instrumentation requires monitoring to be deployed in advance and the system to remain observable throughout execution. Such conditions are difficult to satisfy in production environments and incident-response scenarios. Volatile memory, in contrast, provides a reliable source for recovering the actual runtime state of a running application without requiring prior instrumentation. Therefore, this paper presents MEM-SBOM, the first memory forensics framework that generates SBOMs directly from the runtime state of Python applications. It recovers the modules from the interpreter's internal structures, resolves package versions, and analyzes bytecode to build dependency graphs and identify vulnerable functions. We implemented MEM-SBOM as a suite of Volatility 3 plugins and evaluated it against 51 real-world Python applications. It achieves 100% extraction accuracy, identifies Streamlit as the only application that calls the vulnerable routines of the tornado dependency, and recovers all runtime packages missed by existing SBOM tools, providing more accurate dependency graphs and better vulnerability assessment. These capabilities make MEM-SBOM a practical foundation for software supply chain security and incident response by providing a forensically sound runtime view of what is executed on a system.

### 2. 方法论
- **核心思想**：利用 Volatility 3 对进程内存镜像进行分析，从 CPython 解释器的内部数据结构（如 `sys.modules`、线程上下文、垃圾回收链表、内存分配区）中恢复**全部已加载模块**，并进一步解析版本、依赖和调用链，生成 CycloneDX 格式 SBOM 并进行漏洞可达性分析。
- **关键技术细节**：
  - **多层级模块提取（应对 6 类逃避技术）**：
    - L1：解释器注册表 (`sys.modules`)
    - L2：线程执行栈帧的命名空间
    - L3：进程级 GC 双向链表遍历
    - L4：`pymalloc` 区域扫描，按对象头验证模块类型
    - L5：堆区原始扫描，以 8 字节对齐偏移匹配 `PyObject` 头
  - **模块过滤与分组**：排除内置模块和标准库，识别第三方包，并按包名（第一个点前子串）分组，利用安装路径区分第三方与标准库。
  - **版本解析**：优先读取模块 `__version__` 等属性；若无，则通过 `sys.path_importer_cache` 中的 `.dist-info` 获取安装名，并经 PEP 503 标准化匹配；若仍有冲突，查询 PyPI API 获取 `top_level.txt` 映射后回退本地缓存版本。
  - **SBOM 序列化**：输出符合 CycloneDX 的 JSON，包含组件、PURL、依赖关系及内存溯源元数据。
  - **依赖图构建**（算法 1）：对每个模块遍历其 `__dict__`，识别模块对象（`md_name`）、类对象（`__module__`）和可调用对象（`func_module`），递归处理；同时对函数字节码应用滑动窗口（窗口宽 4）匹配 11 种导入模式（标准、from-import、动态 `exec`/`importlib`/`__import__`），提取跨包依赖边，最终合并成包级有向图。
  - **漏洞检测与可达性分析**：使用 Grype 匹配 CVE，并查询 NVD 和 GitHub Advisory 提取漏洞函数名；然后通过后向污点传播，从漏洞函数追溯至主包入口的调用链，并利用 5 类字节码调用模式（含内部函数、闭包、装饰器等）确定谁真正触发了漏洞。

### 6. 主要结论与发现
- **100% 运行时覆盖**：MEM‑SBOM 能够完整恢复所有安装的、导入的和动态加载的包，甚至在恶意逃避技术下也能保持完整性。
- **曝光版本不一致**：发现了 10 处安装元数据版本与代码中嵌入版本不符的情况（如 `beautifulsoup4` 0.0.2 vs 4.13.4），表明静态 SBOM 可能被误导。
- **精准的漏洞可达性**：在 6 个依赖 `tornado` 的应用中，仅 `Streamlit` 在运行时实际调用并触发了 CVE‑2025‑47287 等漏洞函数，消除了 83.3% 的包级误报。
- **现有 SBOM 工具的不足**：元数据驱动的工具因不支持多种格式（如 PEP 621、Poetry）或忽略未固定版本而失败率高；部署环境分析工具仍遗漏 4–6% 的运行时包，导致依赖图不完整（G_completeness 低于 62%），漏洞 F1 值显著落后（如 CDXGen 仅 23.7%）。

### 7. 优点
- **首创性**：首次将内存取证用于运行时 SBOM 生成，填补了动态语言供应链透明度空白。
- **抗逃避设计**：5 层模块提取可对抗注册表删除、覆盖、GC 操纵等多种高级隐藏技术，取证可靠性高。
- **细粒度关联**：结合字节码分析和污点传播，实现从包到函数级的漏洞可达性验证，大幅降低误报。
- **全面对比**：与 8 种主流 SBOM 工具在静态和部署两个维度上的严格评估，结果说服力强。
- **无侵入性**：无需预先插桩或修改生产系统，仅依赖内存快照，特别适合事件响应。

### 8. 不足与局限
- **语言限制**：仅针对 CPython 解释器，不支持 Python 原生 C 扩展模块的关联分析，无法揭示跨语言边界攻击。
- **瞬态攻击遗漏**：基于快照，无法捕获仅在安装或瞬时执行阶段出现的恶意载荷。
- **依赖已知漏洞数据库**：漏洞检测依赖于 Grype 和公共 CVE/NVD 数据，无法发现零日或未公开漏洞。
- **假设解释器完好**：若 CPython 解释器本身被篡改（如字节码篡改），可能保留有效元数据但行为异常，框架目前未检测此类深层次腐败。
- **版本适配**：需要为不同 Python 版本更新内部偏移量，虽已支持 3.6–3.12，但维护成本随版本迭代存在。
