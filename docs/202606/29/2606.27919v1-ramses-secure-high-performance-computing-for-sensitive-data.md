## 摘要
传统上，高性能计算（HPC）系统的架构针对速度进行了优化，而高度安全的计算机系统则不得不为安全性牺牲速度。然而，生命科学等众多科学领域要求将性能与安全结合起来，以便大规模处理敏感数据。在此，我们介绍RAMSES（增强安全性的建模与仿真研究加速器），这是一个从头设计、在强大安全框架内提供高性能的HPC系统。RAMSES集成了AMD处理器的基于硬件的内存加密，以及来自IBM Storage Scale和Thales CipherTrust管理器的先进文件加密，建立了一个HPC平台，确保数据在整个生命周期——静态、传输和使用中——持续加密，符合主要数据保护标准（欧洲通用数据保护条例、ISO/IEC 27001认证和联邦信息处理标准）。此外，我们实施了高级操作系统加固、多层安全架构和强制多因素认证，使HPC环境适应更高的安全要求。来自生物医学领域的基准测试结果表明，安全环境对性能的影响有限，并且可以在保持系统一致、灵活且用户友好的同时，实现速度和安全这两个冲突需求的整合。

## Abstract
Traditionally, the architecture of high-performance computing (HPC) systems is tailored for speed, while highly secure computer systems must sacrifice speed for security. However, a wide range of scientific domains, such as the life sciences, call for a combination of performance and security to allow processing sensitive data at scale. Here, we present RAMSES (Research Accelerator for Modeling and Simulation with Enhanced Security), an HPC system designed from the ground up to deliver high performance within a robust security framework. RAMSES integrates hardware-based memory encryption of AMD processors with state-of-the-art file encryption from IBM Storage Scale and the Thales CipherTrust manager, establishing an HPC platform that ensures continuous encryption throughout the data life cycle - at rest, in transit, and in use - in compliance with major data protection standards (European General Data Protection Regulation, ISO/IEC 27001 certification, and Federal Information Processing Standards). In addition, we implemented advanced operating system hardening, a multi-layered security architecture, and mandatory multi-factor authentication to adapt the HPC environment to increased security demands. Benchmark results from the biomedical sector demonstrate that the performance impact of the secure environment is limited and that integration of the conflicting requirements speed and security can be achieved while preserving a coherent, flexible, and user-friendly system.

### 1. 研究动机与核心问题

- **问题背景**：传统高性能计算（HPC）以速度为优先，安全系统则牺牲性能；而生物医学等数据敏感的科学领域亟需同时满足 **高性能** 与 **强安全** 的大规模计算环境。
- **现有局限**：多数认证机构仅依赖组织流程（如 ISO/IEC 27001），可信研究环境（TRE）多只保护数据静态和传输安全；可信执行环境（TEE）虽可保护使用中数据，但在 HPC 中的集成和性能仍有瓶颈。尚未出现覆盖 **静态、传输、使用中** 全生命周期加密的实用 HPC 方案。
- **核心目标**：构建从设计之初即融合硬件加密、密钥管理、多层防御的 HPC 系统，在合规（GDPR、ISO 27001、FIPS）前提下证明高性能与高安全可以兼顾。

### 2. 方法论与技术细节

- **整体架构**：设计 **“安全分区”**，通过硬件内存加密、文件加密与密钥管理三层实现持续数据保护。
- **文件加密（静态数据）**
  - 采用 IBM Storage Scale (GPFS) 文件系统加密 + Thales CipherTrust Manager (CTM) 管理密钥。
  - 每个文件拥有独立的 **文件加密密钥（FEK）**，由 **主加密密钥（MEK）** 包裹；MEK 存储于 Thales Luna K7 HSM 内部且永不出硬件。
  - 计算节点通过 KMIP 协议、TLS 加密通道向 CTM 请求 FEK 的包裹/解包裹，实现透明的文件加解密。
- **内存加密（使用中数据）**
  - 基于 AMD SEV (Secure Encrypted Virtualization) 技术：AMD 内存控制器内联 AES 引擎加密 VM 内存页，宿主机及 hypervisor 无法访问明文。
  - 安全分区节点以 SEV 模式启动，通过 KVM/QEMU 启动 **临时虚拟机**，VM 镜像只读，运行态数据存储于加密 GPFS 卷。
  - VM 内预置 KMIP 客户端证书，透明获取文件加密密钥。
  - 使用 SR‑IOV 将 InfiniBand 网络虚拟功能直接分配给 VM，保障高速存储访问。
- **用户访问与调度整合**
  - **强制 MFA**：SSH 登录需密钥对（ED25519 或 RSA4096）+ Cisco Duo 第二因子（移动 App/硬件 Token/TOTP）。
  - 提供前端密钥生成网站，在浏览器端生成密钥对，公钥通过 LDAP 发布。
  - 自定义 Slurm SPANK 插件：验证用户授权，自动启动/销毁专用 VM，实现单参数提交安全作业，用户无感知切换。
- **操作系统加固与特权管理**：分层安全等级（S0–S3），管理平面采用 **单向连接**、跳板机、严格防火墙，阻断反向访问，最小化攻击面。

### 6. 主要结论与发现

- **安全开销可接受**：全加密场景（VM ＋内存＋文件加密）相对于裸金属基线，I/O 密集任务性能下降仅 **4.4%**，内存密集任务下降 **18.0%**。
- **开销来源解析**
  - 虚拟化是主要瓶颈，贡献了约一半的性能损失（BWA‑MEM2 中虚拟化开销 8.1%，占总体 18% 的 45%）。
  - 内存加密在内存密集型场景中额外开销显著，文件加密几乎无额外影响。
- **系统可行性**：RAMSES 成功将机密计算集成到实际 HPC 工作流中，用户通过单一 Slurm 参数即可启动高度安全的执行环境，无需更改原有代码。
- **安全性提升**：相较于仅依赖认证或仅保护静态/传输数据的方案，RAMSES 实现了 **全生命周期加密**，且符合 GDPR、ISO 27001 等法规要求。

### 7. 方法优点

- **首个完整集成案例**：首次将基于硬件的内存加密（AMD SEV）、HSM 密钥管理（Thales）和文件系统透明加密深度整合到生产级 HPC 环境中。
- **开销透明化**：细粒度消融实验清晰分解了虚拟化、内存/文件加密各自的开销，为后续优化指明方向。
- **用户体验友好**：定制 Slurm 插件和临时 VM 机制，用户无感切换；密钥管理自动化，降低合规负担。
- **多层防御设计**：硬件加密、OS 加固、MFA 和单向管理平面构筑纵深安全体系。
- **开源共享**：提供 Slurm 插件和密钥管理网站代码，促进学术机构复现与改进。

### 8. 不足与局限

- **工作负载覆盖有限**：仅测试了两个单节点生物信息学流程，未涉及典型 HPC 应用（如 MPI、GPU 计算、大规模 I/O），结论的通用性待拓展。
- **虚拟化性能瓶颈**：全 VM 是主要开销来源，论文提及未来计划尝试 Kata Containers 等轻量级隔离，但目前仍缺乏优化方案。
- **平台依赖性**：当前方案强依赖 AMD SEV，未实现跨平台（Intel TDX、ARM CCA）的可移植机密计算架构。
- **管理与运维开销**：每个文件集/用户需独立 VM 镜像及 KMIP 证书，自动化程度仍不足；临时 VM 的日志与持久化机制亦待完善。
- **规模与扩展性未验证**：未讨论节点数量、并行作业扩展性及多用户争用下的安全运维策略。
- **缺乏 GPU 机密计算**：虽有计划但未实现，现有 NVIDIA Hopper 等硬件能力未集成。
