## 摘要
我们提出SBridge，一种精确的方法，用于识别二进制文件中与给定源代码函数相似的函数。识别二进制代码中的重用代码对安全至关重要，特别是检测传播的漏洞。尽管二进制到二进制比较可行，但利用源代码作为参考更为实用，因为源代码更容易收集和直接分析，无需编译。然而，源代码与二进制表示之间存在显著差异，包括函数内联，给跨域函数检测带来了挑战。现有方法主要依赖字符串字面量或整个函数之间的结构相似性，无法捕获详细的代码行为，导致大量误报。SBridge通过一项关键创新解决了这些局限：基于控制块的函数匹配，通过将函数分割为条件语句和循环等有意义的单元，封装了基本的功能特征。利用控制块作为跨域表示，SBridge能够精确衡量源代码与二进制代码之间的函数相似性，有效克服函数内联和二进制符号剥离带来的挑战。为进行评估，我们从BinKit收集了3,904个实际C/C++二进制文件。在识别与输入源函数相同的二进制函数的实验中，尽管约40%的二进制函数被内联，SBridge仍实现了75.13%的recall@1和80.98%的recall@5，优于现有方法所能达到的最高43.31%的recall@1和50.2%的recall@

## Abstract
We present SBridge, a precise approach for identifying functions in binaries that are similar to the given source code functions. Identifying reused code in binaries is critical for security, particularly for detecting propagated vulnerabilities. Although binary-to-binary comparison is feasible, leveraging source code as the reference is more practical because source code is easier to collect and analyze directly without compilation. However, significant gaps between source and binary representations, including function inlining, create challenges in cross-domain function detection. Existing approaches primarily rely on string literals or structural similarities between entire functions, failing to capture detailed code behavior and generating many false alarms. SBridge addresses these limitations through a key innovation: control block-based function matching, which encapsulates essential functional features by segmenting functions into meaningful units such as conditionals and loops. Leveraging control blocks as a cross-domain representation, SBridge enables precise measurement of function similarity between source and binary code, effectively overcoming challenges posed by function inlining and stripped binaries. For evaluation, we collected 3,904 real-world C/C++ binaries from BinKit. In experiments identifying binary functions identical to input source functions, despite approximately 40% of binary functions being inlined, SBridge achieved 75.13% recall@1 and 80.98% recall@5, outperforming existing approaches, which achieved up to 43.31% recall@1 and 50.2% recall@

### 1. 论文的核心问题与整体含义

- **核心问题**：如何精准地从目标二进制文件中找到与给定源代码功能相同的函数（源到二进制函数相似性检测）。软件复用日益普遍，尤其是商业现货（COTS）软件中大量使用开源组件，准确检测复用代码对于漏洞传播分析、许可证合规等安全实践至关重要。
- **研究背景与挑战**：
  - 二进制到二进制比较虽可行，但维护编译变体库成本高，以源代码为参照更自然。
  - 源到二进制匹配面临三大挑战：
    1. **源码上下文丢失**：编译会删除变量名、数据类型等信息，且多数二进制文件抹除了符号表。
    2. **架构与编译器多变性**：不同指令集架构、编译器、优化选项导致二进制表示差异巨大。
    3. **函数内联**：编译器内联优化将多个源函数合并到一个二进制函数中，打乱原有调用边界。
- **现有方法局限**：依赖字符串字面量或函数整体结构相似性（如B2SFinder、BinaryAI）无法捕获细节行为，对内联处理不足，导致大量误报。源到源匹配方式（如MRT‑OAST 比较反编译代码与源码）在语法差异大时精度也不高。

### 2. 论文提出的方法论

SBridge 核心思想是 **基于控制块的跨域函数匹配**，将函数分解为语义单元（控制块），通过块的相似度来度量函数相似度，以克服内联和表示差异。

#### 2.1 控制块定义与提取（P1）

- **控制块**：函数内部封装关键功能特征的最小语义单元，如条件语句、循环、函数调用、字符串字面量等。
- **七种块类型**：
  - C1 条件块（if、switch），关键特征为条件表达式，块内容为条件体内代码。
  - C2 else条件块，无关键特征，仅有块内容。
  - C3 循环块（for、while），与条件块类似。
  - C4 字符串块，直接比较字符串字面量。
  - C5 标准C库函数块（346个标准C函数及替换函数组），关键特征：(函数名, 参数个数)，块内容为参数信息。
  - C6 用户自定义函数调用块，同上。
  - C7 递归函数调用块，同上。
- 提取时考虑**内部分支**（条件/循环）和**外部分支**（跨函数调用），并进行层次化提取以处理嵌套结构。
- **归一化**：移除注释、字符/十六进制转十进制、统一将变量名替换为LVAR/PARAM，减少编译引起的变化。

#### 2.2 控制块相似性测量（P2）

- **内部分支块（C1～C3）**：构建特征向量，包含：
  - 静态部分：是否使用局部变量、参数变量；操作符分组（14组）；346个C库函数是否出现；唯一用户函数计数匹配。
  - 动态部分：字符串、数值、调用函数等动态元素。
  - 使用余弦相似度分别计算**关键特征向量**的相似度 `∅_k` 和**块内容向量**的相似度 `∅_b`，最终块相似度 `Φ_b = (∅_k + ∅_b)/2`。
- **外部分支块（C5～C7）**：若函数名和参数个数匹配，`Φ_b=1`；否则检查参数信息（值或类型），给出 1/0.5/0。
- **字符串块（C4）**：直接比较，`Φ_b∈{0,1}`。

#### 2.3 基于块的函数匹配（P3）

- **初步函数相似度**：对每个源块，在候选二进制函数中寻找相似度 ≥ 阈值 θ_b（设为0.7）且最高的二进制块，一对一匹配。`Φ_f = |{被匹配的源块数}| / |{源函数总块数}|`。
- **处理函数内联**：
  - 利用源侧调用图进行深度优先搜索，判断当前函数是否可能有子函数被内联到二进制函数中。
  - 若分析表明**高概率内联**（源函数有子调用但二进制函数无外部调用块），则长度权重 `w_length=1`，不因二进制函数变长而惩罚。
  - 若低概率内联且二进制块数显著多于源，则用对数加权 `w_length = 1.0 / (1.0 + log(#blocks_b / #blocks_s + 1))` 降低相似度，避免误匹配完全不同的大函数。
- **最终函数相似度** `Φ = w_length * Φ_f`。依此对候选二进制函数排序，输出最相似的top-k。

### 3. 实验设计

- **数据集**：来自 BinKit 的二进制代码相似性分析基准，选取其中二进制文件最多的两个GNU软件包：**Coreutils (v9.0)** 与 **Inetutils (v2.4)**，共122个包。
- **编译配置**：4种架构（ARM32, ARM64, x86, x64）× 2种编译器（GCC 9.4, Clang 13.0）× 2种优化（-O0, -O2）× 2种符号状态（strip, no_strip），共32种环境，产生 **3,904个二进制文件**，含624,192个函数。
- **评价指标**：Recall@1, Recall@5, MRR（平均倒数排名）。由于每个源函数只针对二进制的前k个候选评估，若正确函数未进入则计为FN，FP与FN重合，故仅报告Recall。

- **新颖的控制块粒度**：将函数抽象为跨域稳定的语义单元，直接面向编译后变化最小的结构，解决了源码与二进制表示鸿沟。
- **内联处理机制**：通过源侧调用图分析和长度自适应加权，使内联场景下函数匹配显著优于现有方法。
- **特征设计精巧**：基于操作符分组、库函数替换等价类等归一化方法，减少了编译选项差异带来的扰动。

- **实验全面、对比客观**：覆盖多种编译环境，与源‑源和源‑二进制两类先进方法对比，并展示漏洞检测应用潜力。

### 8. 不足与局限

- **依赖反编译质量**：SBridge使用Ghidra生成伪代码，若反编译器在激进优化下失败或生成错误结构，将直接影响控制块提取和匹配准确性。
- **优化覆盖不完整**：实验仅用了‑O0和‑O2两种优化级别，未包含‑O1、‑O3等，虽作者认为‑O1被‑O2覆盖，但实际‑O1可能表现不同。
- **数据集代表性有限**：仅选用了BinKit中二进制数量最多的两个软件包（Coreutils和Inetutils），可能不足以代表所有C/C++项目，泛化能力有待进一步验证。
- **极端代码变形仍误匹配**：在‑O2下对代码逻辑进行大幅修改的情况有时会导致错误映射，即便使用长度权重，超长函数也可能产生误匹配。
- **漏洞检测方法较简**：漏洞检测仅采用块覆盖得分比较，对修改小的补丁可能产生较多假阳性，F1分数仍有较大提升空间。
- **块覆盖比例下降**：部分简单赋值语句未纳入控制块，二进制反编译产生大量无意义赋值虽不利用但亦未说明是否带来信息损失风险。
