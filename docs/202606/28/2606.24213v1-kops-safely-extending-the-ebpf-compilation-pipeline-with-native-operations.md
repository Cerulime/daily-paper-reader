## 摘要
eBPF 能够在网络、可观测性和安全等领域安全地扩展操作系统内核。这种安全性源自内核中的编译流水线：验证器对每个程序进行检查，内核即时编译器（JIT）将验证后的字节码转换为本地代码。内核为保持可信性而让 JIT 保持简单，通过单趟编译每次只翻译一条字节码指令。这种单趟设计错失了优化机会，根据我们的性能表征，eBPF 的运行速度最多比本地编译代码慢两倍。直接在内核 JIT 中添加优化需要获得上游认可和漫长的发布周期，会扩大可信计算基（TCB），并增加各架构的内核代码量。为了解决这些问题，我们提出了 Kops，这是一个扩展接口，允许用户态编译器和内核模块在不修改内核核心的情况下引入新操作，同时保持最小化的可信计算基（TCB）。每个操作都有两种形式：一组普通 eBPF 指令构成的证明序列（由现存的验证器检查），以及 JIT 编译时发出的机器指令原生序列。由于验证器检查的是证明序列，因此原生序列是每个操作对 TCB 的唯一新增部分。硬件惯用操作是该接口最容易获取的优化目标。借助 Kops，我们构建了 EInsn，包含七种操作（如旋转和条件选择），这些操作均可由 CPU 以单条指令执行。Lean 4 证明表明，每个原生序列的计算结果与相应的证明序列相同。在 x86-64 和 ARM64 上，EInsn 将 eBPF 微基准测试的性能最高提升了 24%，生产应用最高提升了 12%。同样的接口还支持整个程序的原生替换，在 TCB 增大的代价下，性能提升可达 2.358 倍。

## Abstract
eBPF safely extends OS kernels in domains such as networking, observability, and security. The safety comes from an in-kernel compilation pipeline where a verifier checks every program, and a kernel just-in-time compiler (JIT) translates the verified bytecode to native code. The kernel keeps the JIT simple to stay trustworthy, translating one bytecode instruction at a time in a single pass. This single-pass design misses optimization opportunities, so eBPF runs up to twice as slow as natively compiled code in our characterization. Adding optimizations to the kernel JIT directly requires upstream acceptance and a long release cycle, enlarges the trusted computing base (TCB), and grows the per-architecture kernel code. To address this, we present Kops, an extension interface that lets userspace compilers and kernel modules introduce new operations without modifying the kernel core, while keeping a minimal trusted computing base (TCB). Each operation has two forms, a proof sequence of vanilla eBPF instructions that the existing verifier checks and a native emit of machine instructions that the JIT compiles. Because the verifier checks the proof sequence, the native emit is the only per-operation addition to the TCB. Hardware idioms are the lowest-hanging fruit for this interface. With Kops, we build EInsn, seven operations such as rotate and conditional select that CPUs execute as single instructions. Lean 4 proofs show that each native emit computes the same result as its proof sequence. On x86-64 and ARM64, EInsn speeds up eBPF microbenchmarks by up to 24% and production applications by up to 12%. The same interface also supports whole-program native replacement, reaching 2.358x at the cost of a larger TCB.

- **主要矛盾**：直接在内核 JIT 中引入优化会扩大可信计算基（TCB）、增加内核代码量且需要长周期上游合入，而现有的用户态优化器（如 K2、Merlin、EPSO）只能工作在 eBPF 指令集内，无法识别和生成原生硬件指令。

## 2. 方法论

### 2.1 Kops 扩展接口核心思想
- **双形式操作**：每个新操作包含两条路径：
  - **证明序列**：一组普通 eBPF 指令，供内核验证器进行安全检查。
  - **原生发射**：对应架构的机器指令，由 JIT 直接编译并执行。
- **安全边界**：验证器只检查证明序列，因此原生发射是操作引入的唯一新增可信代码；识别器、证明序列本身均不在 TCB 内。
- **流水线改造**：在验证前由加载时对扩展指令进行**降低**，替换为证明序列；验证通过后再**恢复**原扩展指令，供 JIT 发射原生代码。

### 2.2 扩展指令编码
- 每条扩展指令由一对现有 eBPF 操作码组成：`BPF_CALL`（标识描述符）和 `MOV64_K`（携带 52 bit 压缩操作数），通过预留的 `src_reg` 值区分。

### 2.3 内核模块注册与 JIT 分发
- 操作以内核模块形式注册，通过 BTF 和 kfuncs 机制暴露描述符。
- JIT 在遇到扩展指令时，根据描述符调用架构自适应的原生发射，若无对应原生代码则回退到证明序列。

### 2.4 EInsn：七个硬件习语操作
- 基于 Kops 实现了 **EInsn**，针对 JIT 编译效率差的硬件惯用操作（如旋转、条件选择、位提取、字节交换、预取、批量拷贝、地址计算），每个操作均有对应架构的原生指令（如 x86-64 的 `ROL`、`CMOV`；ARM64 的 `ROR`、`CSEL` 等）。
- 所有 EInsn 操作的等价性已通过 **Lean 4** 形式化证明：原生发射与证明序列在所有 eBPF 可见的架构状态（寄存器、内存）上结果一致。

## 6. 主要结论与发现

- **性能恢复显著**：EInsn 在微基准上加速 **1.24×（x86-64）** 和 **1.22×（ARM64）**，分别恢复与原生差距的 42% 和部分份额；生产应用中 Cilium 吞吐量提升 **7.4%**，Katran 提升 **7.3%**；代码尺寸显著缩小（~23% 在 x86-64）。
- **安全保证不变**：Kops 不修改验证器核心分析，仅增加通用的降低/恢复机制，原生发射是唯一新增 TCB 部分，且已由 Lean 4 证明与证明序列等价。
- **策略指导**：并非全部启用最优；例如 Cilium 禁用 bulk_memory 后吞吐进一步提升（+11.4%），而 Katran 增加覆盖反而导致性能下降，说明需要基于工作负载选择操作族。
- **全程序原生替换上界**：在信任整个原生代码的前提下，性能可达 **2.358×**，远远超过 EInsn 收益，但 TCB 显著增大。

## 7. 优点

- **安全性与性能的优雅折衷**：通过证明序列 + 原生发射的双形式设计，在不改变验证器核心的前提下引入硬件优化，TCB 增量极小。
- **严密的等价性证明**：使用 Lean 4 形式化验证每个 EInsn 操作的原生发射与证明序列结果一致，提供了机器可检查的安全证据。
- **架构无关且易于扩展**：操作以内核模块注册，识别器和证明序列与架构解耦，新架构只需提供原生发射即可获得优化。
- **实际效果验证扎实**：同时评估纯计算基准和真实生产数据面程序，并给出策略选择的实践指导，避免了对覆盖率本身的盲目追求。

## 8. 不足与局限

- **优化范围有限**：仅针对少数硬件习语（旋转、选择等），无法解决跨指令优化（如寄存器分配、指令调度）带来的性能损失，仍有一部分性能差距不可恢复（约 58%）。
- **策略调优依赖人工**：目前不同操作族的启用/禁用需要人工针对工作负载做选择，缺少自动化的 cost-model 指导，可能出现不合时宜的回归。
- **形式验证覆盖不全**：Lean 4 证明仅覆盖 EInsn 操作的语义等价，但不涵盖缓存、分支预测等微架构效应；全程序原生替换实验则未给出形式证明，只是典型的 trust but verify 模式。
- **解释器支持缺失**：Kops 扩展指令不适用于纯解释器路径，若内核无 JIT 则直接拒载程序，限制了其适用场景。
